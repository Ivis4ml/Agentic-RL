AutoGym：RL 训练环境的自动生成与验证

agentic RL 里稀缺的不再是生成器，而是可信、已验证、可归因的环境。AutoGym 是一条闭合回路的 pipeline：它生成 RL 环境，并验证其中哪些可被信任用于训练，且把二者耦合起来，使验证信号引导下一步生成什么。它的贡献是这条回路本身，即一个定向生成策略加上使该策略 fitness 诚实的信任层，而非任一半边单独成立，因为裸生成正在商品化，而孤立的验证只是一道闸。

团队：LLM Core & Agentic AI  ·  状态：迭代草稿  ·  范围：coding / agentic / 长程 RL 环境  ·  日期：2026-06  ·  版本：v1.8
一段话论点

AutoGym 是一条闭合回路的 pipeline，它做两件事并将其耦合：生成 RL 环境，并验证其中哪些可被信任用于训练，且让验证信号把生成引向真正能推动策略的那些环境。任一半边单独都不新：裸环境生成在两类 substrate 上都在商品化，即 bash/file/test（coding）与 tools/stateful-DB/grader（business），而孤立的验证只是一道闸。贡献在于这条回路，确切说是一个其 fitness 为受治理、可信验证器（而非可被操纵的 proxy）的生成回路，这恰是决定以往 self-play 与 meta-generation 系统是提升还是崩溃的关键。具体而言：AutoGym 采用开源生成器（AWM、EnvFactory、SWE-rebench）与 NeMo Gym 的组合基底，作为置于其自有 binding 无关 Core IR 之后的组件，并自建使回路闭合的两个半边，即一个定向生成策略（朝受训者当前弱点与 projected transfer 生成，而非追求裸吞吐）以及使其 fitness 诚实的信任层（分层验证器分类、reliability auditing、环境证书、以及 agent-as-artifact provenance）。进展以 transfer per unit of human effort 衡量，而非裸环境数。第一垂直是 coding/SWE，因为其奖励是干净的程序 oracle；business/tool-use 垂直作为难验证情形随后，那里 ground 在真实有状态系统使验证半边成为决定性因素。

01 — 动机

稀缺资源是可信环境，而非裸生成

在 2025–2026 的文献里，同一个结构性事实反复出现：造出一个候选环境是廉价的，造出一个其奖励信号可信的环境是昂贵的，而良率正是在这里坍塌。微软的 MAI-Thinking-1 报告给出了我们手上最硬的数字。其内部 SWE pipeline 从 1.02 亿个公开 GitHub PR 出发，保留约 487 万个带链接 issue 的，其中 42.8% 构建成可执行容器，15.3% 通过 reference-grading 信号抽取，最终只有 5.5%（265,617 个环境）通过环境与 grader 验证[1]。要点不是某一阶段主导了数量流失（构建阶段实际剔除得最多），而是可执行性是一道廉价、弱的过滤：成功构建的环境里只有 12.8%（208 万中的 26.6 万）能通过 grading 与验证两道闸，且每个幸存候选的边际成本集中在构建之后（两遍测试 grading、多次 golden/empty 复验、非确定性过滤、低质 spec 重写）。决定奖励可信度的，是这道闸，而非裸生成。对一条自动环境 pipeline 的含义是结构性的：因为廉价生成与稀缺可信是同一漏斗的两端，你无法只自动化其一而不顾其二，生成必须与验证耦合，而这正是 AutoGym 自建的回路。

互补的发现在价值一侧：transfer 来自真实性与多样性，而非裸数量。EnvFactory 仅在 85 个 ground 于真实在线工具生态的环境上训练，便胜过用了约 5 倍合成环境的既有工作，在 BFCLv3 上把 Qwen3 提升最多 +15%[2]。Corecraft 从质量方向给出同一论点：单个手工构建、带专家 rubric 的环境，在一个 GRPO epoch 后即带来对 Toolathlon 的 +6.8% 分布外 transfer[3]。所以真正要紧的两个量，即verified yield 与下游 transfer，都由验证与 grounding 质量支配，而非生成器吞吐。

推论

大量生成环境是这套 stack 里廉价、已商品化、已开源的部分。研究机会在于把廉价生成转化为可信训练供给的那条回路：朝策略真正薄弱处生成，判定其中哪些环境足够可信到可用于训练，加以证明，并把该判断回送以引导下一轮生成。生成与验证不是两个项目，而是同一条 pipeline 的两个半边。

02 — 既有工作

全景：生成拥挤，验证稀疏

这一领域可按环境来自何处（其 grounding 来源）干净地排序，因为这条轴同时决定了多样性/真实性与验证器的难度。我们把已有工作归为五种范式：前三种是生成，第四种是组合，第五种是 meta-optimization。

A — 从真实工件采集

可得的最干净奖励。SWE-rebench / SWE-rebench-V2（Nebius）是把真实合并的 GitHub PR 转成逐任务 Docker 环境、以测试套件为奖励的开源 pipeline[4]；MAI-Thinking-1 的 SWE pipeline 明确"受 SWE-ReBench 启发"，并加入 fail-to-pass / pass-to-pass（F2P / P2P）reference-grading、"时间回溯"式 git 清洗、以及 test-file 重置以阻断 reward hacking[1]。TermiGen / terminal-bench-env（UCSB）把同一思路推广到 11 个类别下的 3,500+ Docker 环境，全部归约为"agent 发出 shell 命令；奖励 = 跑测试"[5]。Qwen3-Coder-Next 的报告把同一 pipeline 陈述为规模化的前提：大体量 agentic 训练需要合成可验证任务并配以完全可执行的环境（其第一条合成线即把 GitHub PR 挖掘成可运行环境），外加高吞吐运行它们的执行基础设施[35]。由于动作空间（bash）通用、奖励是真实测试套件，这一范式几乎把验证器风险消除殆尽，这也是把 coding 作为第一垂直的核心理由。

B — Grounded 合成（代码构建的有状态世界）

tool-use substrate 上的主流路线。AWM（Agent World Model，Snowflake）从 100 个种子名合成 1,000 个 SQL 支撑的 MCP 环境；每个平均 18.5 张表、35 个工具（约 2,000 行代码，FastAPI + SQLAlchemy + MCP），分两阶段构建（先 schema，后代码）以抑制长代码幻觉，并由 code-augmented LLM-as-a-judge 评分[6]。EnvFactory（华为诺亚 / HKUST）在 grounding 上走相反方向：一个三 agent 回路（Search → Code → Test）从真实在线来源发现工具，构建可 Pydantic 序列化的有状态 DB 与 MCP 接口，对其施加四项执行检查校验，并在依赖工具图上采样多轮轨迹[2]。EnvScaler 从既有任务集构建；AgentScaler 与 AutoForge 依赖预收集的工具或文档[2]。APIGen-MT（Salesforce xLAM）是数据导向而非环境构建，但其可验证蓝图（一个 ground-truth 动作序列加预期终态，由 LLM 委员会校验）是一种干净的 oracle 侧验证模式[7]。Corecraft（Surge AI）是手工构建而非 pipeline，但它是本组里最强的 transfer 证据，也是 state-grounded 分解 rubric 的参考[3]。FunReason-MT、SWE-Smith、BugPilot、SWE-Mirror 等线（MAI 引用）在那些能构建但未过质量检查的环境之上提供合成问题[1]。EnterpriseOps-Gym（Zhang 等，2026）值得标注为有状态性的极端：164 张 DB 表、512 个工具，且状态跨 episode 持续，经 Lee taxonomy 综述[8]（二手来源）报告，故这些数字在成为承重论据前应对原始来源确认。DeepSeek-V3.2 是这一范式的前沿闭源实例：一条合成 pipeline，产出数千个代码构建、带程序化奖励的可执行环境，pipeline 与环境均未释出[6,33]，其报告携带本提案作为既有工作所倚的两条消融。其一，难度由模型 solve rate 度量：在其合成任务样本上，合成模型自身 pass@1 仅 12%，而最强前沿模型达 62%，确立了合成任务可被标定为难[33]。其二，纯合成 RL 可迁移：仅在合成 agentic 任务上训练，便在 τ²-Bench、MCP-Mark、MCP-Universe 上带来可观提升[33]，这是"来自完全合成 substrate 的 transfer 真实且可测"的最强公开证据，也是 Phase 0 合成 cell 所依的前提（§10）。

命名说明 — 补充

已存在一个发表过的 "AutoEnv"：Zhang 等（2025），"AutoEnv: Automated Environments for Measuring Cross-Environment Agent Learning"[31]，一个以约 每环境 4.12 美元生成异构 agent 世界、用于构建跨环境学习 benchmark（AutoEnv-36，36 个环境 / 358 关）的自动框架。那与本提案是不同问题：一个用于度量 agent 如何跨环境学习的 benchmark 与环境生成器，而非带验证器治理的 RL 训练环境供给链。我们用 AutoGym 以回避命名与文献撞名。

C — Self-play / 协同演化（补充，此前未在此综述）

第三种范式：策略自身以 proposer/challenger 角色生成任务分布。Absolute Zero（AZR）训练单一模型以"低但非零"的成功率提出 coding 任务，并用 Python 解释器验证，零外部数据[9]。R-Zero 跨多个 reasoning 领域协同演化 challenger 与 solver；SPIRAL 用零和博弈；Multi-Agent Evolve 把协同演化推广到多样任务[10]。具体到 SWE 领域，Self-Play SWE-RL（SSR）在真实沙箱仓库上，于 bug 注入 agent 与 bug 求解 agent 之间运行对抗回路[11]；GASP（Guided Asymmetric Self-Play）把 proposer ground 向一个指定的、真实数据的难目标集，而非自由探索[12]。Agent-World 最接近一个自演化的环境竞技场：它探索真实工具生态，合成难度可控的可验证任务，再运行"诊断弱点 → 生成定向任务 → 重训"回路[13]。整条线源自经典 RL 中的 unsupervised environment design（UED / ACCEL）与基于 regret 的自动课程[12]。

关于范式 C 的长期警告

"Zero" self-play 受限于模型既有知识与固定的环境规则；未 ground 的单种群优化倾向于坍向平庸。有效的方法都加入了 grounding（SPICE 的语料、SSR/GASP 的真实仓库）或对一个移动目标的对抗协同演化。AutoGym 应把 self-play 视为喂入同一套验证器与证书闸的高方差来源，绝不作为绕过它们的无监督捷径。开放的经验问题（Agent-World 的审稿人已指出）是 freeze-environment-growth 消融：增益有多少来自更难的任务，有多少来自学习信号本身[13]。

D — 组合 / 编排：NeMo Gym

NeMo Gym（NVIDIA，Apache-2.0）是组合层，也是本提案最重要的外部依赖。它的模块布局，即 resources_servers/（每个环境 = tools + state + verifier + 配置 YAML + tests）、responses_api_agents/、responses_api_models/，由 Ray 上的 ng_run / ng_collect_rollouts 驱动，正是我们想要的分解，且它以 OpenAI Responses API（而非 MCP）作为 agent↔model↔env 协议的标准[14]。训练侧它集成 NeMo RL、OpenRLHF、Unsloth、VeRL；环境侧吸纳 Reasoning Gym、Aviary、OpenEnv、Verifiers；附带一个用于接入的 .claude/skills/add-benchmark Skill；并已在 Nemotron 训练中经受实战[14]。关键在于，它的环境目录已以出货形态枚举了完整的验证器分类（见 §05）。

E — 生成器的 meta-optimization：SkillOpt

SkillOpt（Microsoft）是一个文本空间优化器，把 skill 文档当作一个冻结 agent 的可训练状态：一个独立的优化器模型把打过分的 rollout 转成有界的增/删/改编辑，仅当其严格提升一个 held-out 验证分时才接受，采用 step/epoch 两时间尺度更新[15]。它解决的是"自动改进一个生成 harness"的优化器；它没有解决fitness 函数，而对环境生成而言，后者正是验证器问题本身。见 §09。

Part B — 验证与 provenance：被低估综述的一侧

上表之所以密集，是因为这一领域把力气集中在了生成。如今这一点在视野内的每个前沿配方上都重复出现：MAI[1]、DeepSeek-V3.2/V4[33,34]、Qwen3-Coder-Next[35] 都在规模上出货合成 pipeline 与沙箱基础设施（仅 DeepSeek 的 DSec 就运行数十万个并发沙箱[34]），却没有人出货验证器治理或证书层。它们之中最接近者值得精确陈述：DSec 维护一份全局有序的逐沙箱轨迹日志，其声明用途包含 fine-grained provenance 与 deterministic replay[34]，但那证明的是执行（哪些命令跑了、哪些状态变了），而非奖励可信：没有验证器分层、没有 reliability audit、没有覆盖构建与奖励 pipeline 的证书、没有从 requirement 到 spec 到 reward 的 lineage。前沿的 provenance 止步于本提案贡献开始处的下一层。相比之下，本贡献所倚的文献散落在彼此鲜少互引的不同社区，且无一被应用于 RL 环境合成。两条线要紧，它们映射到贡献的两个半边。

Line 1 — 奖励可信（验证器分类与 Tier-2 reliability audit 的理论锚，§05）。Gao、Schulman 与 Hilton 确立：把一个 proxy 奖励优化过头会依 Goodhart 定律退化 gold 奖励，且 proxy 与 gold 的 divergence 随优化压力规律性地变化[18]。这恰是 Tier-2 reliability audit 在经验上测量的（proxy = judge，gold = oracle 或人工标注），而 distribution-aware 再审是对 overoptimization 的在线检测，在策略已访问状态分布上进行，把 §05 从一条工程启发式重新 ground 为对一个已知现象的测量。judge 一侧，Zheng 等记录了使"仅看 transcript"的 judge 可被操纵的具体失效模式：position、verbosity、self-enhancement 偏置，以及有限 reasoning[19]；这些正是 Tier-3 禁令与 code-augmentation 要求所针对的。RewardBench 是为 reward 与 judge 模型打分的 meta-evaluation[20]，是 EnvFamily reliability audit 的自然对比基线。而 PRM 一线（Lightman 等，过程级验证，奖励每一步而非最终答案）[21]，是 Tier-2 以分解、逐判据、state-grounded 评分取代整体式判定的渊源。

Line 2 — provenance 与 attestation（证书与 lineage 模型的锚，§08）。软件供应链框架早已把签名的、可验证的工件 provenance 形式化：in-toto 证明一个工件由谁/由什么、经哪些步骤产出[22]，SLSA 定义构建 provenance 完整性的分级[23]。C2PA 为内容来源做类似之事[24]。Datasheets for Datasets 与 Model Cards 是训练数据与已训模型的文档化对应物[25,26]。结构上，AutoGym 的证书 + provenance→agent lineage 模型，是面向 RL 环境的 SLSA 式 provenance，并以奖励可靠性审计加以扩展：EnvFamily / TaskSet / EpisodeReplay 是覆盖环境构建与奖励 pipeline 的签名 attestation，而 EnvFamily 证书是一个环境 family 的 datasheet。

贡献超出既有 attestation 之处

主动陈述此点，因为审稿人否则会提出：SLSA / in-toto 证明一次构建如声称般发生；它们对一个奖励信号在学习策略下是否可信或可复现只字未提。本贡献补上了供应链 attestation 从不需要（因其工件是静态的）的两样东西：(1) distribution-aware 再签名：证书针对当前策略的已访问状态分布被重新校验，而非一次性签发，因为威胁是一个学会利用验证器的策略，而非被篡改的构建；(2) 奖励可复现容差带：Tier-2 奖励仅在固定 judge 快照下、于某个容差内可复现。一句话定位：面向 RL 环境的 SLSA，外加静态 attestation 从不需要的奖励可信机制。这正是生成表所显现的缺口：每一行都解决了生成，但我们未发现任何开源系统同时具备以下四者：closed-loop directed generation、verifier reliability auditing、environment certificates、transfer attribution。单个行各自出货局部（可执行任务、replay 日志、质量过滤、一个 verified 子集），但没有人出货这个组合，而该组合的既有工作散落于两支文献，据我们所知本提案是首个为环境合成将其汇合者。

03 — 定位

AutoGym 是一条架于开源组件之上的回路，而非一个新 framework

从分层后退一步看：搭一条自动环境 pipeline 是四件事，其中只有两件是开放问题。去哪找 case 与如何做大在开源界正在商品化，AutoGym 采用之。该信任哪些 case 与如何引导下一步造什么尚未解决，AutoGym 自建之，而它们正是回路的两个半边。（下方的 stack 按层展示同一切分；这里是按任务的同一切分。）

ADOPT · 去哪找好 case

grounding 来源：采集的真实工件（SWE-rebench、TermiGen）、grounded 合成（AWM、EnvFactory）、self-play（AZR、SSR）。正在商品化，置于 IR 之后采用。我们保留的唯一判断及其边界：realism 胜过 volume[2]是一种逐环境优势，而非前提，纯合成代码构建任务在前沿规模上同样可迁移[33]。§02。

ADOPT · 如何做大

生成吞吐是生成器的问题；rollout / 训练规模是已知工程，即面向并发的沙箱路由与一套开源 RL rollout 栈。是配置，不是研究。§06 · §12。

BUILD · 如何过滤不可信 case

承重的一半，且"坏"是三件不同的事：构建失败 / flaky（廉价的 TaskSet 检查）、奖励不可信（验证器分类 + 受治理 Tier-2 judge + reliability audit）、以及不迁移（attribution）。第一件是廉价闸；后两件是贡献。§05 · §08。

BUILD · 如何定向生成

事后过滤即"多造、再扔掉大部分"，其成本随体量增长。把信任 + transfer 信号回送，使下一批朝着可信且高 transfer 的方向生成，便把供给规模转化为有用供给的规模。§09。

这两个自建的半边是定向生成策略与验证层；策略实际产出的是一个 EnvSpec（§04），即一个环境的声明式描述，也是人类从 requirement 编写出的同一工件（图 8），由 L0 构建、L2 实例化为可运行的 Core IR 实例。因此一轮读作requirement 或弱点分析 → EnvSpec → build → IR → rollout → verify → certify，信任信号回闭到策略上。下方静态 stack 展示哪些层是采用的、哪些是自建的；回路是从信任层（L5）回到生成（L0）的反馈边，见 §08–§09。

这不是收窄野心，而是把工作放到回路上。生成器已商品化且在开源界持续改进；它们都未出货的，是引导生成的回路，以及使其 fitness 诚实的信任层。两个半边都是贡献：一个针对受训者弱点与 projected transfer 的定向生成策略（§09），以及决定一个奖励何时可信、并使所得能力可复现、可归因的验证器合同 + provenance 模型（§08）。二者不可分割：没有可信 fitness 的定向生成就是 meta-reward-hacking（会坍塌，§09），而没有生成的验证只是一道静态闸。NeMo Gym 做了轻量的逐环境 license 追踪与 tests[14]，但没有定向生成回路、没有验证器分类、没有 reliability auditing、没有 provenance-to-agent lineage，而这正是本项目所承担的范围。

由构造保证低摩擦

一个控制平面只有在"认证比绕过它更便宜"时才值得存在。认证必须作为 CI 中对每个环境的自动检查运行，而非一道人工把关步骤，即一条一行命令的 autogym certify 路径，并对尚未通过的环境给出明确状态（可用于实验，排除于可信训练混合之外）。让治理层被忽略的，是摩擦，而非能力。

IR 独立性，强制执行而非口头声明

NeMo Gym 明确处于早期、API 仍在演进[14]。我们把它作为 L1 的一种实现，置于一个接到我们自有 Core IR（L2）的 adapter 之后。一个只对单一 backend 编写的 adapter，不过是改了字段名的那个 backend，所以 IR 独立性被设为一条退出条件，而非一条原则：该 IR 必须绑定两个组合 backend，即 NeMo Gym 与 OpenEnv（AWM 的基础设施已并入 OpenEnv[6]，故第二个绑定几乎是免费的）。在它绑定两个之前，L2 边界只是名义上的。

04 — 抽象

Core IR：Context + Tool，而非 MCP

我们关心的两类 substrate 表面看不同：coding agent 驱动终端与文件系统；business agent 对一个有状态数据库调用 typed tools。但它们共享一个抽象。一个环境向 agent 暴露一个 context/observation、一个架于可变 world backend 之上的 action/tool schema，并由一个 out-of-context grader 评分。MCP 是该 action schema 的一种线缆编码，而非抽象本身。

这不是理论，而是前沿实验室实际的构建方式。MAI-Thinking-1 对 SWE 与 tool use 都跑同一个 ReAct 回路：SWE 中动作是 bash 与一个字符串替换编辑器，tool use 中是对带种子数据库的 typed 调用，而 tool schema 用的是 OpenAI function-calling schema，而非 MCP[1]。NeMo Gym 以 OpenAI Responses API 为标准[14]。AWM 与 EnvFactory 用 MCP[2,6]。三个前沿系统，三种不同 binding，这正是抽象 binding、而非锁定其一的论据。

该抽象的运行期实现，即一个被编译并绑定到 backend 的 EnvSpec，我们称之为 Env/Gym：一个 substrate 无关的接口，每个被构建的环境都遵从它，因此不论底层世界是代码容器还是 SQL 数据库，trainer 看到的都是同一份合同。

reset(instance)  -> observation                        // 起沙箱、置初态、返回 agent 可见观测
step(action)     -> observation, terminated, truncated  // terminated = 任务结束；truncated = 预算/超时截断
grade()           -> {verified: r, preference: r'}        // channel-tagged；preference 永不并入 verified（§05/§07）
// truncated episode 被标记并排除评分、不记为失败；runtime 截断不得被读成 reward-hacking（§05；Phase 0 divergence 指标）
// dual-control 变体（如 τ²-bench）：一个配对的 UserGym，user-simulator 也 reset/step 共享世界

这正是 NeMo Gym 与 Harbor 已暴露的接口[14]，所以给它命名只是固定了一条生态早已遵守的边界：EnvSpec 是源形态；Env/Gym 是运行期工件，而"一致格式"意味着 §04 表中的每个来源，不论 substrate，都编译到同一套 reset / step / grade 合同。§04 的 Core IR 就是这个 Env/Gym，只是从 binding 一侧描述；二者是同一对象的源态与运行期两个名字。

为什么这是正确的边界

bash 接口把逐领域的工具合成问题归约为零，你只需合成 world + verifier。typed-tool 环境仍需逐领域工具，但 trivially container-free。两者都从一个带声明式 action schema 与 binding 的 IR 中落出。

泛化出来的是什么

不是"coding"本身，而是agentic skill。TermiGen→SWE 与 Corecraft→Toolathlon 的 transfer 表明多步分解、工具探索、约束处理可跨表面迁移[3,5]。coding/terminal 是最佳训练 substrate，是因为其奖励干净，而非因为该技能是 coding 特有的。

一种编写格式：EnvSpec（种子版）

上面的 IR 是运行期形态，即 agent 与 trainer 所面对的。它还有一个源形态，即环境被构建自的统一输入：一份声明式 EnvSpec，描述一个任务 family 的 context 表面、action/tool schema、world backend、grounding 来源，以及（闭合回路的那部分）其验证合同（verifier tier、grader、anti-hacking 姿态）与 provenance 字段。这条 pipeline 是 compiler 形的：编写或生成一个 EnvSpec → 一个 builder 将其物化 → 它被实例化为一个 Core IR 实例 → 绑定到一个运行期。spec 是生成半边所产出、验证半边所认证的；trainer 消费的是实例化后的 IR 实例，而非 spec 本身。这在所引工作中已现端倪：AWM 的两阶段 schema→code[6]、NeMo Gym 的 resource / add-benchmark 声明[14]、APIGen-MT 的任务蓝图[7]，而 AutoGym 的新增是把验证合同与 provenance 折入同一份 spec。这也把 §04 焊到 §08：由于 in-toto 式 attestation 附着于工件，而 EnvSpec 就是该工件，EnvFamily 证书签名 (EnvSpec, builder 版本) 这一对。

一个 EnvSpec 描述一个 family，而非一个 task，这正是 grounding.selector 与 grader 的 tests_from 是规则而非字面值的原因。builder 的第一项工作是把 family 展开为一个 TaskSet，即一个 episode 实际运行的具体 instance。两种展开模式，由 substrate 固定。采集型 family（这个 Tier-1 SWE 例）枚举：selector 解析到范围内的真实 PR，每个 PR 成为一个 instance，携带自身的 issue 文本、base commit、链接的 F2P/P2P tests、以及 held-out golden patch。合成型 family（下面的 Tier-2 SQL 例）生成：builder 跑 schema→code，逐 instance 产出 (初始 DB 状态, 请求, 预期终态) 三元组。无论哪种，family 级字段（tier、anti-hacking、action schema、binding）在整个 TaskSet 上共享，而逐 instance 字段（哪个 bug、哪些 tests、哪个初态）各异；随后 RL 从 TaskSet 采样 instance，每 episode 一个，而该采样器本身就是一种课程，注意回路在两种粒度上运行：family 级（生成什么，§09）与 instance 级（replay 哪些 instance，PLR 的主场[28]）。这条 family → TaskSet → episode 粒度，正是 §08 证书在每一级所签名的：EnvFamily 证书签 (EnvSpec, builder)；TaskSet 证书签实例化后的 instance 集（dedup、decontamination、难度覆盖）；EpisodeReplay 签单次已评分运行。这串里有一个术语需要一个定义，否则全文凡用到处都欠它：难度是一个被测量的量，而非一个凭信念贴的标签，即一组固定参考模型 panel 的 solve rate（每 instance n 次 rollout 上的 pass@k）。这正是 SWE-rebench-V2 难度标签背后、以及 DeepSeek-V3.2 对其合成任务标定背后的机制（同一样本上，合成模型 pass@1 为 12%，前沿模型至多 62%）[4,33]。其成本是 rollout，计入 verifier compute（§08）；且它是策略相对的：固定 panel 给出一个稳定的准入期标签，而课程的实时信号是受训者自身的 solve rate，后者随策略改进而漂移，这正是 PLR 式 replay 所针对的非平稳性[28]。

从 spec 到被训练的环境：数据进，过滤出

一个 EnvSpec 是规则，不是一个环境。两件事把它变成环境。其一，构建时注入真实数据，即 selector 的 PR、合成的 DB seed、task 文件，把 family 的规则变成具体 instance（候选 TaskSet）。其二，一道过滤移除无法被训练的 instance。候选 TaskSet 不是可训练的环境；认证后的那个才是。

真实数据从何而来，是它自己的三阶段前端，位于 builder 上游。Crawl 获取原始工件，即 GitHub PR、终端会话、业务日志与工具 trace。Process 把它们变成可构建单元：容器化、固定依赖版本、抽取奖励信号（SWE 中把一个 PR 解析为 issue + base commit + 链接 tests；SQL 或对话中抽取或合成 schema、工具集与 policy）。Filter 丢弃低质，其具体做法因任务而异，但跨 substrate 共享共同模式：golden-passes / empty-fails 检查、按日期 decontamination、dedup、难度与质量标签，这正是下图所绘的闸逻辑。对 coding 而言，这套前端在很大程度上已公开：SWE-rebench-V2 出货约 3.2 万个已爬取、已处理、带质量标签、覆盖 20 种语言并附预构建镜像的任务[4]，故 crawl 与 process 是采用而非重建。对 business 或内部 grounded 的 substrate，crawl 与 process 需要数据团队支持（内部系统，外加隐私与 licensing），这一依赖首次成为承重项是在 business 垂直（Phase 1），而非 coding。Phase 0a（SWE）今天就数据无阻；Phase 0b 的 τ² dual-control cell（§10）自带 Gym 与混合 grader，是带真实搭建成本的那部分。

种子版，而非冻结的标准

该格式起于这个最小种子，并随垂直接入而经验性地精炼。它刻意不预先定稿：过早把环境格式标准化，正是你最终得到一份没有生成器真能产出的 spec 的原因。种子的存在是为了让回路可运行，而非作为最终接口。

spec 之上：人类 requirement 向下编译到它

一个 EnvSpec 精确到足以据此构建，这意味着人类通常不直接写它。人类提供的是意图：一段松散的、requirement 级的描述（场景、所针对的能力、何为成功、硬约束、一个 grounding 提示），再由一个 LLM requirement-to-spec 编译器填入 substrate、action schema、world backend 与 grader 接线。人类说造什么、何为正确；模型说如何组装。同一个 case-resolution family，以 requirement 而非 spec 陈述：

{
  "goal": "train an agent to resolve policy-bound customer case requests",
  "scenario": "support agent with read access to a records DB and a resolution-policy
               document; must decide eligibility and apply the correct resolution",
  "capability_targeted": "multi-step tool use under a policy constraint",   // 可选的引导提示
  "success_criteria": [                                  // 这就是 eval，用业务语言表述
    "the resolution matches policy for the record's state",
    "no resolution applied when policy forbids it",
    "agent cites the policy clause it applied"
  ],
  "constraints": ["read-only on records except the resolution field", "no external network"],
  "grounding_hint": "real resolution policy + record schema with ambiguous / exception edge cases"
}

刻意地，这个 schema 与 EnvSpec 毫不相像。若 requirement 携带 substrate、binding 或 world_backend.type，它就成了一份逼着人类手工填的 EnvSpec，翻译层也就失去意义；两个 schema 形状不同，正是它们做不同工作的证据：一个捕获业务意图，另一个是构建目标。而 success_criteria 正是验证合同的来源：人类陈述"正确"何意，编译器决定如何自动判定它，"the resolution matches policy"编译为一个 Tier-1 状态 oracle；"cites the policy clause it applied"是软判据，编译为一个 Tier-2 分解 judge。验证器合同由此锚定于人类对成功的定义，而非凭空生成，这正是 Tier-2 软判据所要求的。

编译器是信任链的第一环，它也必须被审计

由于 success_criteria 是由一个 LLM 编译到 verifier tier 与 grader 的，一次误编译（把"the resolution matches policy"接到一个草率的状态检查，或把一条已陈述约束从 decomposed_criteria 中悄然丢弃）会在出生时就败坏验证合同，位于 §05/§08 每一道审计的上游。证书签 (EnvSpec, builder)，但目前没有任何东西证明 requirement→EnvSpec 的保真度。所以编译器作为 agent-as-artifact 受治理（§08），并携带一道廉价的保真检查：对人类编写的 family，把编译出的 verification_contract 回读给 requirement 作者签字确认；结构上，做一次 round-trip 反编译并比对（从 EnvSpec 重新生成一份 requirement 摘要，并与原文 diff）。没有这一步，信任链就有一个未经审计的第一环。

05 — 脊柱

验证器分类

这是整个平台的承重墙。早先的设计草案曾把 LLM-as-judge 一概逐出 verified tier。那一立场被那些恰被引为 verified 环境权威的系统所反驳，它必须被一份分层合同取代。这一重构立足于文献反复浮现的 false-positive / false-negative 权衡。

纯 oracle — 低 FP，高 FN

程序化的终态 / 测试检查几乎无法被操纵，但脆：瞬时失败、边角情形、部分执行都会被记为失败。在 RL 中一个 false negative 惩罚正确行为并毒化学习信号，AWM 正是基于这一风险来论证其 judge[6]。

仅看 transcript 的 judge — 高 FP

一个 LLM 读一段对话并打分是灵活的（低 FN），但可被操纵：策略学会满足表面线索。这正是必须留在 verified tier 之外的形态。

每个展示出 transfer 的系统所用的合成都相同：代码抽取 state-grounded 证据，分解 judge 在该证据上推理。AWM 的"code-augmented LLM-as-a-judge"比较数据库前后，抽取变更记录与诊断，再据此判定[6]。Corecraft 把每个任务分解为细粒度、客观、state-grounded 的判据，各自独立评分[3]。MAI-Thinking-1 对 SWE 用可验证测试奖励，同时对合成 tool-use 环境用一个分解子任务 LLM judge，外加对 tests 无法规定者用 AI-feedback 奖励[1]。NeMo Gym 的目录并排出货两种形态，甚至同一领域两种形态并存：Spider2-Lite 是基于执行的 text-to-SQL，奖励为结果集等价的二元判定且明确无 judge，而 Text-To-SQL 用一个 LLM-as-judge 等价检查；Math-With-Judge 把 math-verify 与一个 LLM judge 结合[14]。

治理 Tier 2

把 judge 提升进 verified tier，只有在源系统未出货的治理下才安全。AutoGym 增加三样。(1) 一道judge-reliability 审计，测量 judge 的 false-positive 率，但注意其引导问题：一个任务处于 Tier 2 正因为不存在完整程序化 oracle，故审计集分两部分：(a) 那些确实有完整 oracle 的任务子集，廉价地用来标定 judge–oracle 一致度；(b) 真正软判据的人工标注样本，其标注成本是实打实的，并显式计入验证成本核算（§08），而非藏在"curation"里。(2) 分解、state-grounded 的判据，连续奖励等于被满足断言的比例（Corecraft 式），取代 AWM 粗糙的 1.0 / 0.1 / 0.0 整体分[3,6]。(3) distribution-aware 再审。对静态集测得的 false-positive 率，并不能约束训练策略在其学会利用 judge 时、其分布下的 FP 率。修法复用文档已为 EpisodeReplay 准备的机制：把 Karten 式再签名扩展到 judge 审计，在当前最强策略的已访问状态分布上重新校验 judge 可靠性[17]。adversary 协同演化是它的补充，而非替代。judge 的 rubric prompt 本身是一个带版本、可优化的工件，NeMo Gym 已对 judge prompt 做版本（Terminus "rubrics v4" vs "simple"）[14]，这接到 §09 的 SkillOpt 回路。

preference 通道，一条不同的轴，而非第四个 tier

三个 tier 测量同一件事：一个任务的正确性有多可验证。但 AutoGym 采用的配方还用到了根本与正确性无关的奖励。MAI 对"tests 无法规定者"用 AI-feedback 奖励（任务解读、helpfulness、轨迹质量）[1]，而 Nemotron-3 的第 3 阶段是带 principle-following GenRM 的 RLHF[14]。这些不是关于正确性的 Tier-3 transcript judge；它们是正交轴上的 preference/alignment 信号（helpfulness、style、safety）。因此合同声明一条独立的 preference 通道：在训练中合法可用，永不混入 verified-reward 核算，并在 EnvBlend 中作为一条独立、不可替代的奖励流携带（§07）。没有它，verification_contract 甚至无法表示 §07 作为既有工作采用的 Nemotron 配方，而一份无法表达它所引配方的合同终将被忽略。

冻结 judge 与训练 judge，合同必须声明的第二条轴

上述 tier 默默假定 Tier-2 judge 是一个冻结工件：一个带固定 rubric 的 pinned 快照，每版审计一次、可永久 replay。该领域正朝另一方向移动。GenRM 一线训练 judge，即 Nemotron 的 principle-following GenRM[14]，以及 DeepSeek-V4 的 generative reward model，其中 actor 模型自身充当 judge，且该评判行为与生成一道用 RL 优化[34]。一个训练过的 judge 改变的是失效模式，而不仅是实现：judge–policy 协同适配可能超出任何固定审计节奏，而 self-judging 配置在构造上就让生成器与评估器共享盲点。它还打破 EpisodeReplay 前提：一个持续更新的 judge 没有可 pin 的快照，故其奖励不可 replay。因此合同显式声明这条轴：

Judge 体制进 verified tier？所需治理
冻结快照（AWM 式 code-augmented judge[6]）是 — Tier 2reliability audit + distribution-aware 再审；快照 pin 在 EpisodeReplay 中
训练型，逐训练轮快照（GenRM 一线[14]）是 — 逐快照每轮 checkpoint 须重过 reliability audit，其奖励方计为 verified；证书 pin 该轮快照
持续更新，无 pinned 快照否仅 preference 通道，奖励不可 replay，故 verified-tier 可复现性无定义
Self-judging（actor = judge，V4-GRM 式[34]）否仅 preference 通道，reliability audit 预设 judge/policy 独立，而 self-judging 在构造上消除了它

一句话立场：训练型 judge 只能以被审计的快照身份进入 verified tier，被 verified 的从来不是"那个 judge"，而是一个通过 §05 审计并 pin 在证书中的具体 (checkpoint, rubric, decoding) 三元组，与冻结 judge 完全一样。两次审计之间移动的一切都是 preference 通道信号。一条定价注记：GRM 一线声称的卖点是监督的经济性，V4 报告称仅用极少量人工标注即可从 rubric 引导的数据获得稳健评分[34]。在本合同下，这份经济并非免费：judge 训练所省下的任何标注，reliability 审计都会在准入期重新引入，并显式计价于 §08 成本核算，标注从 judge 移到它的审计，不会消失。

MAI 的 anti-reward-hacking 清单是 Tier 1 与 Conformance 层的具体防御基线：网络访问控制、对 base 之后任何 commit 的"时间回溯"git 清洗、以及带隐藏 test diff 的 test-file 重置[1]。

06 — 运行期

沙箱与运行期：按威胁模型隔离

最常见的单一架构错误，是假定评测拓扑能迁移到训练。它不能。一个 eval harness 在几百个任务上大致每任务跑一次尝试；那里 pod-per-instance 加 Docker-in-pod 没问题，因为 pod 启动延迟会在一段长 rollout 上摊薄。RL 训练需要每步 10²–10³ 个并发 episode、持续数千步，那里启动成本主导 wall-clock。所以 eval 与 train 是两条运行期路径，隔离强度应是谁写了被执行的代码 × eval-还是-train 的函数，而非垂直的函数。

两个实现细节使训练路径可控。其一，沉重的 Python 依赖被预取到置于 base 镜像之外的 venv/conda overlay，Nemotron 的 SWE 容器构建与 SWE-MiniSandbox 的 conda 版本分层是同一招，把 base 保持小、逐任务搭建压到一个 venv overlay[16]。其二，在 container-free 训练中，对 reward hacking 的唯一结构性防御是 Conformance 纪律，外加把 verifier 与 agent 工作区放在不同沙箱运行；这一点在此比在完整容器隔离下更要紧。

本节的生产级存在性证明是 DeepSeek 的 DSec[34]：一个统一 API（单一 Python SDK），覆盖四种按隔离成本排序的执行 substrate，即 function call、container、microVM（Firecracker）、完整 VM（QEMU），在单一集群上运行数十万个并发沙箱，substrate 间切换只是改一个参数。对上面的路由有三点推论。(1) 谱系获得最轻的一档：DSec 的 function-call substrate 把无状态调用 dispatch 到一个预热池，消除逐调用冷启动[34]；对一个可信、合成的纯 Python tool 世界（AWM 式 FastAPI + SQL 进程），同一逻辑延伸到完全不带沙箱的进程内执行。无论哪种，地板都落在 container-free 之下，路由矩阵应为 Tier-2 合成世界路径提供它。(2) 启动延迟有第三种缓解：除 container-free 执行与 venv overlay 外，分层按需镜像加载（DSec 挂载 3FS 支撑的只读 EROFS 层）在不放弃容器的前提下攻击同一成本[34]，恰在 TermiGen 式异质依赖那一堆逼你用容器处相关。(3) Replay 双重承重。DSec 为每个沙箱维护一份全局有序轨迹日志，报告点名其用途：抢占后快进（replay 缓存结果，绝不重执行非幂等命令）、fine-grained provenance、以及任意历史会话的 deterministic replay[34]，正是 EpisodeReplay 证书所需的同一确定性机制（§08），已作为前沿规模的基础设施出货。且该需求有一个超出效率或信任的训练正确性面向：V4 的 rollout 服务观察到，从头重生一个被中断的请求在统计上是错的，较短的响应更常在抢占中存活，故重生会把数据偏向短序列，这正是它改为从 token 粒度 write-ahead log 续跑的原因[34]。一个确定性 replay 的奖励，就是一段免费续跑的 rollout，同时是一份在抢占下保持无偏的数据分布。

07 — 课程

EnvBlend：簇内混合，跨簇分阶段

实际出货的配方比"全混在一起"更微妙。Nemotron-3 Super 跑三个 RL 阶段，即一个跨约 21 个环境、37 个数据集（math、code、STEM、safety、agentic、reasoning-gym）的多环境 RLVR 阶段，再到 SWE-RL（OpenHands + Apptainer），再到带 principle-following GenRM 的 RLHF[14]。第 1 阶段大量混合；Nemotron 在 RLVR、SWE、RLHF 之间的分阶段边界是 infra 驱动的，那些簇需要不同的基础设施与不同的奖励机制，但不应被过度解读为"分离从不助益学习"：下面第二个出货配方出于学习侧理由而分离。

细化

一个 EnvBlend 在一个 infra/reward 同质的簇内混合，并在运行期或 verifier 类型不同时跨簇分阶段。这纠正了"全混胜过分阶段"的过简读法，并把课程直接系到 §06 的沙箱路由决策。把 Nemotron 的 blend schema（rlvr / swe / rlhf family）作为 EnvBlend 工件的既有工作采用，并把其生命周期（版本、隔离检疫、退役）作为控制平面的一部分治理。blend 必须把 §05 的 preference 通道（RLHF / GenRM）作为一条独立、不可替代的流携带，永不并入 verified reward。

第二种出货消费模式：先专家，后蒸馏

DeepSeek-V4 根本不跨领域混合：独立的领域专家（如 mathematics、coding、agent、instruction following）各自在自身同质供给上以 SFT + GRPO 训练，再经来自十余个 teacher 模型的 on-policy distillation 合并（在 student 采样轨迹上做 reverse-KL，全词表 logits）[34]。动机是学习侧而非 infra 侧，报告称该 OPD 合并规避了 mixed-RL 与权重合并路线中遇到的性能退化[34]，且合并发生在蒸馏空间，那里不再消费环境。对 AutoGym 而言这是定位的一个特性，而非矛盾：认证供给与消费者无关。同一批认证 family 既可喂 Nemotron 式混合 blend，也可等同地喂 V4 式逐专家 pipeline；后者中 EnvBlend 工件退化为每专家一个同质 blend，而蒸馏阶段消费的是轨迹，不是环境。因此 §07 规则有其范围：簇内混合 / 跨簇分阶段描述的是一个策略应如何消费混合供给，而是否要只训练一个策略，是 trainer 的配方选择，位于供给层上游、且对供给层不可见。

范围决策 — 责任人：business 垂直

跨 episode 持久状态（EnterpriseOps-Gym 维护 164 表 / 512 工具，状态跨 episode 携带[8]）不是一个可归档了事的边角情形。对企业运营工作流（带持久跨会话状态的有状态业务领域）而言，持久状态可能是核心特性，而非极端。IR 是否在 v1 支持跨 episode 状态，是由该垂直的实际需求驱动的决策，而非由 IR 便利或 per-episode-reset 的 replay 模型驱动；在此标出，供垂直责任人在 §05 的 EpisodeReplay 确定性假设被锁定前解决。

08 — 贡献（1/2）

回路的信任半边：一个奖励何时可信

回路有两个半边。本节是决定何为可信的那一半，即生成半边（§09）被其引导的 fitness 函数。它有两重：决定一个奖励信号何时可信，以及使所得供给可归因、可复现。生成器在开源界正在商品化；这个信任层无一系统已解决，而它正是使生成回路不坍塌为 meta-reward-hacking 的东西。

验证最难处，也是最有价值的环境所在。EnvFactory 的结果（ground 于真实、有状态系统的环境比更大体量的合成环境更易 transfer[2]）意味着最高价值的供给，恰是缺少干净程序化 oracle 的那部分供给（真实系统有软成功判据、部分状态、副作用）。所以信任层不是对有价值环境征的税，而是使它们根本可用、并使回路朝它们而非背离它们生成的东西。

把贡献表述为可证伪的断言

下面的工件（verifier 合同、证书、provenance、auditing）是系统贡献，将开源发布。研究贡献是一组实验可反驳的断言，在此陈述，使本提案可凭"什么会证伪它"来评判。

H1 · 受治理 judge 产出 oracle 质量信号

一个受治理 Tier-2 judge 在无完整程序化 oracle 的任务上提供改进策略的训练信号，在有 oracle 可比的子集上达到与 oracle 训练所得改进的同等水平（在策略改进 Δ 上、于一个预注册的非劣性边界 δ 之内），且不诱发 false-positive 驱动的 reward hacking。证伪条件：在 oracle 子集上，受治理 judge 的改进低于 oracle 超过 δ；或出现 oracle 臂没有的 reward-hacking 坍塌。

H2 · 再审更早检测到被利用

distribution-aware 再审比静态审计更早检测到 judge 被利用，可度量为 false-positive 率相对训练步数的漂移。由于利用未必在一次运行中自发出现，测试在诱导利用条件下进行（一个被刻意削弱的 judge，或一个对抗微调的策略），以保证该现象存在以供检测。证伪条件：在诱导利用下，静态与 distribution-aware 审计在统计上不可区分的步数处报警。

H3 · attribution 是廉价的

family 粒度的 transfer attribution 可在每轮 k 个 family 的 O(k) 额外训练成本内可行，而非精确 leave-one-out 所要求的 k+1 次完整运行。证伪条件：没有任何廉价估计量能在可接受误差内追踪 leave-one-out attribution。

H4 · 回路胜过开环生成

闭合回路（以验证信号引导生成，即受训者弱点 × projected transfer，并以一个可信 fitness 把关）带来比同等体量开环生成更高的每 GPU-hour transfer，且不像 proxy 把关的 self-play 那样坍塌。这是双核断言：回路，而非任一半边，才是贡献。证伪条件：在等算力下定向生成与开环持平；或尽管有受治理 fitness 仍坍塌；或受治理臂 (c) 在 proxy 把关臂 (b) 未坍塌时也未能胜过 (b)，即治理对回路毫无增益（§10，Phase 2）。

证书模型

EnvFamily 证书

grounding 来源的 provenance 与 license、构建确定性、verifier tier 及其 reliability audit、anti-hacking 姿态。没有它，任何 family 都不得进入可信 blend。

TaskSet 证书

一个 EnvSpec 展开成的 instance 集（§04）。签名 golden-passes / empty-fails 检查、对 held-out eval 的 decontamination、难度与多样性覆盖（难度按 §04 定义：参考 panel 的 solve rate）、泄漏风险。

EpisodeReplay 证书

确定性 replay 与可复现奖励，针对当前策略的已访问状态分布再签名[17]。对 Tier-2 奖励，可复现性要求 pin 住 judge 模型快照 + decoding 参数 + rubric 版本；对随机奖励，证书携带一条 replay 容差带而非精确匹配。judge 弃用按设计使证书失效，而非悄然失效。

Provenance → agent lineage

每个可信环境、task、reward 都可追溯到签名它的生成 agent 版本与 verifier 版本。这使一次训练运行可复现，使你能把一次能力变化归因于导致它的供给，并通过 replay 其 lineage 来 debug 一次回退。

Agent-as-artifact

生成、修复、验证环境的那些 agent，即把 success_criteria 接成验证合同的 requirement-to-spec 编译器（§04），以及在 dual-control 环境中的 user simulator（一个 UserSimSpec），它们本身就是带版本、lineage 可追踪的工件（AgentSpec + AgentVerifier + Scorecard）。编译器与 user simulator 列入此名单的理由与 verifier 相同：二者都是奖励决定者。一次误编译在每道审计上游败坏合同；一个糟糕的 user simulator 可使任务无解或泄漏答案，这是 τ-bench 一线有详细记录的噪声源，故二者受同样治理。这是 meta-generation 层（§09）所优化的 substrate，而它在所综述的每个 framework 中都缺席。

验证的经济学

验证成本并不均匀，预算必须分配，而非平摊。三类成本是 verifier compute（容器执行、多次复验，逐次便宜，随 rollout 数量增长）、人工标注（Tier-2 审计集，逐任务昂贵，真正稀缺的输入）、以及 adversary 协同演化（周期性，在一个 family 上摊薄）。分配策略由下面的成功度量得出：昂贵的 Tier-2 人工审计被把关给高 projected transfer 的 family，而低 transfer 或纯 Tier-1 的 family 走廉价自动检查。在一个模板化 Tier-1 family 与一个高 transfer grounded family 上花同等验证力气，正是该度量旨在防止的浪费。冷启动告诫：projected transfer 来自 attribution，attribution 需要一次训练运行，运行需要一个认证 family，起点处是一条循环依赖。在任何 attribution 数据存在之前的引导先验，是 projected transfer 的一个廉价 proxy：grounding 真实性、多样性覆盖、以及与目标 benchmark 的领域接近度。

如何度量进展

进展以 transfer per unit of human effort 报告，即价值比成本，而非单看成本。价值是一次训练运行中可归因于某环境 family 的 held-out 能力变动（如 SWE-bench-Verified Δ）；成本是抵达一个 verified 环境的力气，三个分量被显式列出以免任何一个被隐藏：人工分钟（含 §05 的 Tier-2 标注）、GPU-hour、认证算力。路线图的逐 GPU-hour 读数（§10）是同一度量的算力分量，而非一个竞争指标；人工力气仍是头条，因为它是稀缺输入。把价值与成本配对，避免了纯成本度量的失效模式（那会奖励廉价模板化 Tier-1 环境，正是 §01 所警告的"计数而非多样性"陷阱），并封闭了任何"可被操纵的 proxy 不算数"防护中的循环：一个操纵了其 verifier 却未推动 held-out 能力的环境，在价值上得零分，故 transfer 信号本身就是对 gaming 的首要检查，reliability audit（§05）为次要检查。

一个 held-out benchmark，一旦回路朝它优化，就不再是 held-out 的

生成一旦被"对某 benchmark 的 projected transfer"引导，那个 benchmark 就进入了上一层的优化，且回路会经分布收敛对它行 Goodhart（生成一份模仿该 benchmark 任务统计的训练混合），这是 §05 overoptimization 结果的 meta 级形态，是 instance 级 TaskSet decontamination 检查抓不到的。在回路下存活的度量纪律是一道second-level holdout：回路的 fitness 在一个steering benchmark A 上计算，而 H4 及每个报告的 transfer 数值都在对 fitness 永不可见的held-out benchmark B/C 上读取（如在 SWE-bench-Verified 上 steer；在 SWE-bench-Pro 与 Terminal-Bench 上报告），并跨运行轮换。单层"held-out"对开环臂已足够；只有闭环需要第二层。该现象在生态尺度上已可见：NIST/CAISI 对 DeepSeek-V4 的独立评测发现，在该实验室自家报告未含的 benchmark 上（一个 held-out 软件工程集、ARC-AGI-2 semi-private），结果明显弱于其报告所选 benchmark 给人的印象[36]，无论是否显式闭合了一个回路，实验室自选 benchmark 都表现得像一个 steering 集。

Transfer attribution 是度量的根基，也是一个开放问题

上面每个定量断言都立足于"把 held-out 能力变动归因于某个具体环境 family"：分配策略、anti-gaming 防护、以及 §09 的 meta-generation fitness 都消费它。所以 attribution 不是脚注，它是承重的，而显而易见的方法不可扩展。精确 leave-one-out 每 family 花一次训练运行；每轮 k 个 family 就是 k+1 次运行，在任何真实 family 数下都不可持续。因此研究项是廉价 attribution 估计量，即 grouped / cluster 级消融（hold out 簇而非单例）、从单次运行做 influence-function 式估计、以及 reward-correlation 诊断，并在一个小参考集上对精确 leave-one-out 校验。一个基础设施杠杆直接改善该参考集：确定性、batch-invariant 的训练 kernel。DeepSeek-V4 在训练与推理上出货端到端 bitwise batch-invariant 的确定性 kernel，其动机是 debugging、定位 loss spike 等异常的数值成因、以及一致的 post-training 行为[34]。对 attribution 的推论由我们来下，且很直接：bitwise 可复现的运行从精确 LOO 基线中移除 run-to-run 非确定性，攻击 attribution 三个障碍之一（noise），而留下另两个（non-additivity、drift）不变。这就是上面的 H3，单凭它就是本提案里更具发表价值的开放问题之一：一个在 mixed RL 训练中做 O(k) 成本 family 粒度 credit assignment 的方法，其用处远超本平台。

分工，重述

NeMo Gym 做组合。开源生成器做合成。verifier 库做评分。AutoGym 闭合回路：它引导生成什么、决定什么可信、对其认证、并把该判断回送以引导下一轮，使所得能力可复现、可归因。这句话就是贡献。

09 — 贡献（2/2）

回路的生成半边：定向合成，诚实把关

这是另一半，也是本项目是一条回路、而非把一个 verifier 螺接到生成器上的原因。定向生成策略朝策略当前薄弱处与 transfer 最高处合成，随后验证决定什么存活并把该判断回送。该机制有已出货的既有工作。SkillOpt 提供一个有纪律的优化器：把一个生成 harness 表为 skill 文档，以受 held-out 验证分把关的有界编辑改进，并用 step/epoch 两时间尺度更新[15]。Agent-World 提供协同演化参考：explore → diagnose-weakness → generate-targeted-tasks → retrain[13]。二者都未提供、且使这成为一项贡献而非一次重实现的，是一个你能信任的 fitness 函数。

为何这一半依赖另一半，全提案的承重点

SkillOpt 解决优化器；它不解决 fitness 函数。对一个生成 skill，验证分就是 §05/§08 的 verifier 问题。把定向生成以一个廉价 proxy（verifier pass-rate）把关，就是 meta-reward-hacking，会坍向与未 ground 的 self-play 相同的平庸，这正是回路在没有信任半边时危险、且两个核不可分割的原因。用 SkillOpt 的两时间尺度结构，把廉价 proxy（L0–L2 conformance）放在 step 编辑上，把诚实而昂贵的信号（Tier-1 奖励稳定性 + adversary 存活 + held-out 下游 transfer，§08）放在 epoch 闸上，并协同演化一个 adversary，使 fitness 是一个移动目标而非静态集。

排序：回路是终点，但它按顺序、而非一次性建成。信任半边（§08）必须在回路闭合前稳固，否则定向生成策略会去优化一个它能操纵的 fitness。所以实际的第一形态是接入时的逐类型 harness 适配（一个新 Gym 类型 → 种子 skill → SkillOpt 适配），而非一个单一的全局 meta-agent；完整的跨 pipeline 回路只在存在 2–3 条带 instrumentation 的生成 pipeline 后闭合，每条都产出干净的 Scorecard 遥测与一个 held-out、decontaminated 的 generation-target 验证集。解锁回路的第一个工作项是定义那个逐类型 fitness（即 §08 verifier 合同），而非搭建优化器。

两条上游汇入一份 spec，以及人类上游为何是平等的

以弱点引导生成，预设了弱点可见。自动信号使它可见，但只在一个边界之内。rollout-failure 分布显示策略在已存在的环境上何处失败；transfer attribution（H3）对已训练过的 family 排序；adversary 协同演化在当前环境的编辑空间内搜索。三者都在已有覆盖的分布上加密，而无一能看见一条根本没有任何环境的能力轴：一个新垂直、一种未被代表的工具组合、一类无人为之造过世界的真实约束。那是 regret 式 autocurricula 的结构性盲点，而非调参缺口，信号是定义在既有分布之上的，故它无法指向其外。

所以人类编写的 requirement 是一条平等的上游，而非一根可退役的引导拐杖。人类提供分布外方向，即那个说"这个垂直重要、且无人造过"的领域先验，这是 explore/exploit 权衡的探索一端，而自动 regret（偏向加密已知困难区域）在结构上对其供给不足。两条上游汇于同一 EnvSpec：人类一线是 requirement → LLM 翻译 → EnvSpec（§04）；机器一线是弱点/transfer 信号 → 定向生成策略 → EnvSpec。它们把同一工件产入同一池子，被同等地 build、验证、认证、attribute，故信任与 attribution 机制不以哪条上游产出该 family 为条件。provenance 仍记录它（每个 family 携带其 generator_agent），但没有任何闸把人类编写的 family 与策略生成的区别对待。用 UED 的话说这是一个老张力：regret 式 autocurricula 朝已知难度的前沿锐化（下一小节），而搜索空间本身的广度向来来自 domain randomization 或人类设计的 level 分布。EnvSpec 不过是人类打开空间、策略在其内运行 regret 的那个接口。

它在生成侧文献中的位置

以受训者弱点引导合成，就是 unsupervised environment design 的 regret 式课程：PAIRED 的 minimax-regret teacher[27]、PLR 的 regret 优先 level replay[28]、ACCEL 的 edit-and-evolve learnability 前沿[29]，而 Agent-World 是同一 explore-diagnose-generate 回路的 agentic-LLM 实例[13]。以 projected transfer 引导它则是另一渊源：data-mixture / data-selection 优化，属 DoReMi 式混合 reweighting 一线[30]。AutoGym 的生成半边不属于其中任一单独者，它是 受治理 verifier 下的、transfer 加权的 regret：UED 的难度信号被 §05 信任半边变得可信（使 teacher 无法靠生成无解或误评的 level 取胜），并从裸难度重新指向 held-out transfer。两个引导信号可分离，将被独立消融，即仅弱点（纯 regret，无 attribution）、仅 transfer（纯 data-selection）、以及二者之积，因为它们有不同的失效模式与不同的成本。

把一个 benchmark 放进回路，带来两条诚实约束。其一，回路的 fitness 在一个 steering benchmark 上计算，而 H4 在 §08 的 second-level holdout（B/C）上测量，回路从不看见评判它的那个 benchmark。其二，transfer 加权消费 H3 的廉价 attribution 估计量，故回路有一条言明的降级路径而非悄然失败：若 H3 不成立，定向生成回退到仅弱点引导（无需 attribution）加粗粒度 grouped-cluster 消融，H4 在降低的引导分辨率下仍可测。

10 — 执行

P0 计划与路线图

Coding/SWE 是第一垂直，且是一个严格的去风险选择：奖励是干净的程序 oracle（Tier 1），它复用一条标准 SWE-bench-Verified eval pipeline，并携带最强的泛化叙事。policy-bound case-resolution / SQL 垂直作为难验证情形（Tier 2）随后，那里 oracle-vs-judge 的切分可被廉价证明。

Phase 0 — 先证认证价值，再证外部效度（P0a → P0b）

Phase 0 被排序，使第一发是"能给出有信息量失败的最小之物"。P0a 在最干净的 substrate（SWE，Tier-1）上证明认证有价值，且今天就数据无阻；P0b 加入 τ² dual-control substrate，以检验该结果不是 SWE 的特例，它带真实 infra 成本，故跟在 P0a 之后而非阻塞它。两个子阶段合起来构成 2×2（substrate × filter）：P0a 是闸，P0b 是外部效度。

共享控制平面（两个子阶段）：把 Core IR adapter 架到 NeMo Gym resources-servers 上[14]；出货 EnvFamily + TaskSet + EpisodeReplay 证书 v0；从第一天起就给成本分母（human-minutes-to-verified）做 instrumentation。

P0a — 仅 SWE、Tier-1、certified-vs-raw（先跑）。

  • 采用 SWE-rebench-V2 构建 pipeline，并以 SWE-MiniSandbox / Apptainer 作为 L3 训练运行期[4,16]；eval 保留一条标准 Docker/K8s pipeline。action schema = bash + str-replace；带 MAI anti-hacking 清单的证书 Tier-1 路径[1]。
  • 在匹配的环境数、匹配的难度桶、匹配的 rollout 预算下，训练（GRPO）两个臂：certified = SWE-rebench-V2 带质量标签、按日期 decontaminated、附预构建镜像的子集；raw = 同一 pipeline 中"可构建但未验证"instance 的随机样本（此臂预设可访问 pre-filter 中间产物，须确认公开发布是否暴露它们，否则需预算把开源 crawl/process 阶段在关闭质量闸下重跑一次）。
  • 测量每 GPU-hour 的 held-out SWE-bench-Verified transfer 与 reward-hacking 发生率（test-edit / hidden-test 规避、reward–true-success divergence、按 §04 排除 truncated episode）。需控制的混杂：验证可能过滤掉 flaky-hard 任务，移动 certified 的难度构成；报告难度匹配结果或逐臂分布。读数是臂级的，故 P0a 不依赖于解决 family 级 attribution，那是 Phase-2 的研究项（§08/§09）。

P0b — 加入 τ² dual-control substrate（外部效度，待 infra 就绪）。

  • 采用 τ²-bench 的 AgentGym / UserGym 及其组合式任务生成器[32]；action schema 变为 dual-control（agent tools + user tools 架于共享状态之上）。这是终结"纯 SWE、数据无阻"便利的那一项：它额外需要混合的 Tier-1（DB-state）+ Tier-2（NL-goal judge）grader，以及 EpisodeReplay 中的一个 pinned user-simulator 快照（§08），故其 infra 成本是实打实的，必须预算。
  • 跑 τ² cell：certified = 通过混合 grader 可靠性闸的 family（judge 审计 + pinned user-sim）；raw = 生成但未审计的 family。τ²-bench-Verified 的存在（一次社区再发布，修正了原集中的 task 与评测判据错配）是这条 raw→certified 差距真实而非假想的直接证据。配上 P0a 的 SWE cell，这就补全了 2×2。
Phase 0 退出条件 — 先闸，后交互

P0a 闸：在 SWE 上，certified 臂在匹配的数量与预算下展现出更高的每 GPU-hour transfer 且/或显著更低的 reward-hacking 发生率，这是"验证增加价值"的可证伪检验。若 P0a 失败，议程在此止步；若通过，P0b 即获正当性。P0b 读数 — substrate × filter 交互：若 certified-vs-raw 的增益在 τ²（一个带合成世界、协同行动用户、部分被判定奖励的 substrate）上复现，则信任半边的价值是一种普遍属性，而非 SWE 特例；若它只在 SWE 上出现，则贡献比所声称的更窄，跨 substrate 的框定是错的。（来自合成 substrate 的 transfer 根本可测，这是已确立的既有工作：DeepSeek-V3.2 的纯合成 RL 提升 τ²-Bench、MCP-Mark、MCP-Universe[33]。）合起来，Phase 0 报告的是外部效度，而不仅是一个效应，它在两个 substrate 上检验 H1 的"不坍塌"部分，并为最初的若干 family 引导起 transfer attribution（§08）。

Phase 1 — business / tool-use 垂直（难验证器）

  • 在 IR 之后采用 EnvFactory（真实工具 grounding、依赖工具图）与 AWM（两阶段 schema→code）生成器[2,6]；把 case-resolution / SQL 垂直 ground 在一个真实、有状态的业务领域（那里软成功判据逼出 Tier-2 路径）。
  • 实现带两部分 reliability audit 与 distribution-aware 再审的 Tier-2 code-augmented 分解 judge[14]。
  • 跑头条实验（受治理 judge vs oracle，同领域）：在一个确实存在程序化 oracle 的领域（Spider2-Lite 式执行检查），用 oracle 训练一个臂、用受治理 Tier-2 judge 训练一个臂；再把 judge 臂扩展到 oracle 无法评分的"oracle 不可表达"任务。检验受治理 judge 是否 (i) 在二者皆适用处达到与 oracle 的策略改进同等水平，以及 (ii) 在不出现 reward-hacking 退化的前提下解锁 oracle 不可表达的任务。这就是 H1。
  • 加入 preference 通道；把 MCP 提升为与 Responses API 并列的一等 binding。
Phase 1 退出条件

(a) 双 backend IR conformance，即 Core IR 绑定 NeMo Gym 与 OpenEnv（使 L2 成真的检验，因 AWM 已并入 OpenEnv 而几乎免费[6]）。(b) H1 结果：在共享子集上受治理 judge 与 oracle 持平，外加在 oracle 不可表达任务上的 transfer 增益，且 reward-hacking 发生率不劣于 oracle 臂。(c) 在真实系统 grounded 垂直上、于当前策略分布上审计得到的一个 Tier-2 judge false-positive 率，并有一个可归因于某 Tier-2 认证 grounded family 的 transfer delta。

Phase 2 — 闭合回路（定向生成 + blend）

  • 用一个隔离"受治理 fitness delta"的三臂实验闭合回路：(a) 开环生成；(b) 以廉价 proxy fitness 把关的定向生成（verifier pass-rate，即 Agent-World 式信号[13]）；(c) 以 §08 受治理 fitness 把关的定向生成。(a)→(c) 检验回路是否有用；(b)→(c) 是相对既有定向回路工作的 delta，也是对坍塌断言的直接检验，没有臂 (b)，结果只是再证"定向 > 非定向"，而 Agent-World 已展示这点。
  • EnvBlend 工件 + Nemotron 式 blend schema；簇内混合 / 跨簇分阶段；preference 通道作为一条不可替代的流[14]。
  • 经同一套证书闸纳入 self-play 来源（SWE 的 SSR/GASP）；跑 Agent-World 的 freeze-growth 消融[11,12,13]。
  • SkillOpt 逐类型 harness 适配，以诚实的 epoch-fitness 把关[15]。
Phase 2 退出条件 — 检验 H4

在三个臂中，(c) 在每 GPU-hour 的 held-out transfer 上胜过 (a)，且（承重结果）(c) 胜过 (b)，同时 (b) 表现出 (c) 没有的 proxy 把关坍塌。(b)-vs-(c) 的对照，正是把 AutoGym 与一条架在廉价 proxy 上的定向回路区分开的东西；没有它，回路就是多加了几步的 Agent-World。transfer 在 held-out B/C benchmark 上读取（§08/§09），绝不在 steering benchmark 上。逐阶段的规模与责任人留待一次规划处理；此处把退出条件作为当前最低标准给出。

数据 pipeline，外部 vs 内部，是一条横切依赖

上面每个阶段都消费 §04 的 crawl → process → filter 前端，而该前端是 AutoGym 唯一一个其可行性不在项目掌控之内的部分。对外部来源，它基本已解决：GitHub PR 以 SWE-rebench-V2 的形态预先爬取、处理、带质量标签而来（约 3.2 万任务、20 种语言、预构建镜像、CC-BY-4.0）[4]；terminal-bench[5]与 τ²-bench 生成器[32]覆盖其余 Phase-0 substrate。当需要更新或更宽的切片时，SWE-rebench 的 crawl pipeline 是开源的，可在对源仓库加 license 过滤后重跑。外部数据是一项成本，而非一道阻塞。

内部来源在性质上不同，而非程度上不同，它们才是项目真正可能卡住之处。把一个垂直 ground 在真实内部代码库、工具目录或业务数据库上（正是这一点赋予 business 垂直相对合成世界的价值）需要四样东西，无一是研究问题、无一在贡献之内：(i) 访问，构建与训练 infra 必须穿过内部系统的安全控制抵达它们；(ii) 清洗，内部工件携带 secret、凭据、PII，必须在进入任何策略可记忆的 context 之前移除，这是 anti-hacking 机制必须延伸覆盖的义务；(iii) IP / licensing，provenance 必须记录某 family 训练于内部数据，因为这约束了所得模型可作为什么被分享（EnvFamily 证书 lineage 承载此点，但围绕它的策略是组织决策）；以及 (iv) 一条数据团队工作线，用以大规模 crawl、scrub、process。

内部数据获取是 AutoGym 最可能的外部阻塞

该依赖位于整个贡献的上游：若前端无法以足够的体量与质量供给可 ground 的工件，信任半边就无物可认证，定向生成回路就无物可据以生成，回路在它自己的任何机制运行之前就已供给枯竭。所以诚实陈述 AutoGym 最大的外部风险，不是一个建模风险；而是内部数据的访问、清洗、licensing 构成一条数据团队与组织层面的依赖，它把 Phase 0 公共数据 SWE cell 之后的每个垂直都卡住。缓解是排序，而非机巧：在公共数据上跑 Phase 0（今天就无阻），并把内部数据的访问、清洗、licensing 作为一条显式的并行 Phase-1 工作线启动，而非在集成时才发现。

11 — 诚实

开放问题与风险

  • Verifier 可靠性是全部胜负所在。若 Tier-2 reliability audit 薄弱，受治理 judge 会悄悄重新放进该 tier 本要排除的 false-positive。该审计的引导需要对真正软判据的人工标注（按 Tier 2 定义，无 oracle 存在）；该标注成本显式计入 §08 成本分母，不被隐藏。
  • 数据获取是最可能的外部阻塞。外部数据（经 SWE-rebench-V2 的 GitHub PR）是已解决的成本；内部数据（真实代码库、工具目录、业务数据库）是一条访问 / PII 清洗 / licensing / 数据团队依赖，位于贡献之外，并把 Phase 0 公共数据 cell 之后的每个垂直卡住。若前端跟不上，回路就供给枯竭。缓解是排序（§10）：现在跑公共数据 Phase 0，并把内部数据访问作为一条显式并行的 Phase-1 工作线启动。
  • Tier-2 replay 可复现性。一个 judge 评分的奖励只有在 pinned judge 快照 + decoding 参数 + rubric 版本下才可复现；随机 judge 需要 replay 容差带。EpisodeReplay 证书现已携带这些（§08），但 judge 模型的生命周期管理是一项长期运营成本。
  • 回路可能污染它自己的评测。以 projected transfer 引导生成，会把被引导的那个 benchmark 拉进上一层的优化，回路可经分布收敛对它行 Goodhart，这是 instance 级 decontamination 检查抓不到的失效。缓解是 second-level holdout（§08/§09）：fitness 在一个 steering benchmark 上，所有报告的 transfer 在回路从不看见的 held-out benchmark 上，并轮换。此风险为闭环所特有，对仅验证的框定并不存在。
  • Meta-reward-hacking。对生成器的任何自动优化都继承 verifier 问题作为其 fitness，并能更快地操纵它。缓解是 fitness 诚实（§09），而非更多优化器。
  • 生态锁定 / 频繁变动。NeMo Gym 处于早期、API 在变[14]；双 backend conformance 退出条件（§10）是强制执行的缓解，但它带真实维护成本。
  • Self-play grounding。范式 C 来源高方差，未 ground 时可能坍塌；它们必须与任何其他来源一样通过同一套证书闸，绝不绕过。
  • 跨 episode 状态是一个垂直需求决策，而非 IR 便利决策（§07）：对有状态业务领域它可能是核心特性。在 EpisodeReplay 确定性模型被锁定前与垂直责任人解决它。
  • 引文核验。承重参考文献均已对原始来源核验：生成侧系统（EnvFactory、AWM、MAI-Thinking-1、SWE-rebench-V2、τ²-bench、AutoEnv）以及 §02/§05/§06/§08 所倚的四份前沿报告，即 DeepSeek-V3.2（arXiv:2512.02556）、DeepSeek-V4、Qwen3-Coder-Next（arXiv:2603.00729）、NIST/CAISI 的 DeepSeek-V4 评测，全部核对无误。DeepSeek-V4 的论断已对原始报告确认（§5.2.5 DSec、§5.1.1 GRM、§5.1、§3.3），过程中纠正了两处二手来源的失真（确定性 kernel 言明的动机是 debugging 与数值异常诊断，而非数据消融；agentic 数据进入于 mid-training，而非作为 pre-training 的侧重），这正是"对原始来源确认"纪律为何存在的一个实证。§02 Part A 中若干非承重的 2602/2603 标识符（如 TermiGen、SkillOpt）应在对外流通前再过一遍。
12 — 下一步

接到一个 RL trainer 上

AutoGym 是环境与验证的供给；最直接的下一步是把它接到消费它的 RL 训练栈上，使回路端到端运行，而非止步于认证工件。集成面很小且已规定：Core IR rollout 接口（§04）喂 trainer 的 rollout worker，verification_contract（§05）就是 trainer 调用的奖励函数，而一个认证 EnvBlend（§07）定义训练混合及其逐通道奖励核算（verified vs. preference）。具体而言，这意味着把 IR 绑定到一个开源 RL framework，即 verl / SkyRL / slime / miles / AReaL / NeMo-RL 一类 trainer，作为 GRPO/DAPO 的 rollout 与奖励后端（miles 是 slime 面向生产 MoE 的 fork；AReaL 完全异步的 generation–training 解耦天然契合这条 rollout 密集的回路），并以 §06 的沙箱路由决定 rollout 运行期（训练路径用 Apptainer / container-free）。Phase-0a 的 SWE certified-vs-raw 运行（§10）是第一个集成里程碑：它在一个垂直上演练整条路径（IR → trainer → reward → 证书 → transfer attribution），而一旦该路径跑通，闭合回路（§09，Phase 2）就只是把 attribution 信号路由回定向生成策略的事，而非新基础设施。关于该绑定的两条范围注记。其一，trainer 选型应在吞吐之外权衡确定性、batch-invariant 执行：bitwise 可复现的运行正是使 H3 精确 leave-one-out 参考集可负担的东西（§08）[34]。其二，RL 是第一个消费者，而非唯一一个，认证环境也产出轨迹，而前沿配方已在 RL 上游消费它们（DeepSeek-V4 在 mid-training 阶段纳入 agentic 数据以增强 coding 能力；Qwen3-Coder-Next 同样让其合成任务过 mid-training 以及 RL[34,35]），故同一批认证供给以更低集成成本喂 SFT / mid-training distillation，这也是若 RL 绑定延误时最便宜的回退集成。

13 — FAQ

评审中提出的问题

读者反复提出的问题，附文档当前的回答。凡诚实回答是"这是一个开放问题"处，即如实陈述，而非掩盖。

Q1 · Attribution 在整条链上承重，为何把它留作开放问题？

它在四处承重：价值度量（可归因于某 family 的 transfer）、引导信号（projected transfer）、成本闸（昂贵审计流向高 projected-transfer 的 family）、以及回路 fitness（H4 消费 H3 的估计）。移除它，价值论证即坍塌。它之所以开放，是因为它是一次混合、非平稳 RL 运行中的 credit assignment：干净方法（leave-one-out 重训）是 O(k) 次重训，且被 RL 方差淹没（这是一个地板，而现已在前沿规模出货的确定性、batch-invariant 训练 kernel[34]能直接将其降低，§08）；family 贡献是非可加的（协同与冗余破坏线性分解）；边际价值随策略变化而漂移；而小贡献落到 held-out 噪声地板之下。它是 data attribution / influence functions / data-Shapley 的 RL 类比，这些在监督学习中尚且都是近似，无一能干净地迁移到多步 RL。因此文档把它当作一个可证伪假设（H3），并给出言明的回退：若廉价 attribution 失败，回路降级为仅弱点引导加粗粒度 grouped-cluster 消融（H3→H4 路径，§09），而非悄然失败。

Q2 · "second-level holdout"究竟是什么，如何运行？

first-level holdout 是寻常的那个：不在测试集上训练，在 instance 级做 decontamination（TaskSet 证书）。一旦回路以 projected transfer 朝某 benchmark 引导生成，它便不足够，因为回路能经分布收敛对该 benchmark 行 Goodhart，即生成一份模仿其任务统计、却不复制任何一个任务的训练混合，这是 instance 级 decontamination 抓不到的。second-level holdout 把 benchmark 分为一个 steering 集 A（回路 fitness 只读 A）与报告集 B/C（fitness 从不看见它们）；H4 与所有 transfer 数值在 B/C 上读取，而 A 跨运行轮换。操作上它需要 ≥2–3 个相互独立、难度与领域可比的 benchmark、严格隔离（没有 B/C 的选择信号回流进生成）、以及轮换。微妙之处（也是操作关键）在于 B/C 必须足够独立以抓住 meta-overfitting，又足够相关以测量真实 transfer：太独立则不反映 A 的能力增益；太相关则在 A 上的收敛会抬高 B/C 并掩盖 Goodhart。

Q3 · Tier-2 引导中 oracle 子集可能极小，那么人工标注不就是瓶颈吗？

真实的张力，Tier-2 按定义就是无完整 oracle 的任务，故自然带 oracle 的子集很小。四样东西使它可控。(i) 校准集是构造出来的，而非一个自然比例：构建同时携带 oracle 与软判据的任务（如注入已知正确与已知错误的解），以便即使在生产 family 没有 oracle 处也能测量 judge 可靠性，你校准的是 judge，而非每个任务。(ii) 人类审计一个分层样本，而非全部：judge 预标注，人类在决策边界附近、judge 自相矛盾处、或 judge 与弱信号分歧处复核（active / disagreement 采样）。(iii) 校准按 (judge 快照 + rubric 版本) 摊薄，在所有使用该 judge 的任务间共享。(iv) 成本闸拒绝不值得的 family，一个几乎无 oracle 子集、标注沉重、transfer 不确定的 family，正是成本模型应拒绝接入的。残留是诚实的：对无 oracle 子集且核心判据为软的 family，人工标注是一个不可约的地板，显式计入 §08 成本分母而非隐藏。

Q4 · requirement→EnvSpec：read-back 与 round-trip 当真可行吗，且必须由人来做吗？

这个图景是对的：人类提供意图，一个 LLM 编译器产出 EnvSpec。两道检查各异。Read-back（一个人对编译出的 verification_contract 签字确认）需要人，但它是 family 级、一次性的，故摊薄后便宜，尽管它只适用于存在人类作者处。Round-trip 反编译并比对（从 EnvSpec 重新生成一份 requirement 摘要，并与原文 diff）是自动的，但部分：它抓住粗错（丢掉的约束、错的 tier、结构性遗漏），错过细错（约束在但接到了一个草率检查）。所以人类复核并非普遍必需：人类编写的 family 走便宜的 read-back；机器生成的 family（无人类作者）依赖 round-trip 加一道下游兜底，一份产出可被操纵奖励的误编译合同，终会在 §05/§08 审计中显现为 reward–true-success divergence，故编译器不是单点失效。在规模上，人类复核是定向抽样（一个样本加高 projected-transfer 的 family），而非全覆盖。编译器的误编译率本身是一个经验量，这正是它作为带 Scorecard 的 agent-as-artifact 受治理的原因（§08）。

Q5 · IR 独立性退出条件：执行顺序是否有风险，"conformance"精确指什么？

这一关切的两半都对。执行顺序是风险：把整个系统对着 backend 1 构建、只在最后才让 backend 2 conform，有把 backend 特有假设焊进去的风险，使 conformance 变成一次昂贵的改装。修法是让 conformance 成为一道持续检查，即从早期就在 CI 中对两个 backend 跑一组 canary EnvSpec，而非一道最终闸。关于判据本身：同一个 EnvSpec 在固定策略与固定 instance 集下，应在两个 backend 间产生在一条预注册容差带内一致的奖励分布（与 Tier-2 replay 容差同一概念），而非 bit 相等，因为环境语义（世界动力学 + grader 合同）与 binding 无关，而 binding 只是一种线缆编码。这条"容差内分布一致"就是 conformance 的操作定义。超出容差不是失败而是一个发现：binding 正把语义泄漏进环境（如不同的 tool-timeout 行为改变结果），正是 IR 所禁止的耦合，修 IR/binding，或把该 family 标为 backend 特定。若某 family 在某 backend 上根本无法表达（缺失能力，如没有有状态 DB），则判据收窄到两个 backend 都能表达的 family，而单 backend family 被记为一条 backend 特定的边界，这是对 IR 普遍性的一个诚实限制。

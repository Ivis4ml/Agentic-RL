ProRL-Agent-Server / Polar · 采样到训练样本

从一个黑盒 agent，
拿出干净的token 级训练数据

本页专门拆解一句话背后的具体算法实现。普通 RL 直接调模型就能拿到 token 级数据；而真实 agent 不可控、说方言、多轮、还会失败。Polar 用六个环节把这个难题解决，每一环都给出真实代码与可点击的源码位置。

跑它 + proxy 拦截 + 从推理服务器取精确 token 与 logprob（TITO，不重新分词） + 只在真正采样出的助手 token 上算梯度 + 用状态和占位保住 GRPO 组完整。
00对照

普通 RL 采样 vs 对 agent 采样

先看清差距在哪。切换两种模式：

你完全控制生成循环。一次调用，token 与 logprob 直接返回，无需重建。

# 概念化：标准单轮 / 多轮由你自己写的 workflow 控制
resp = engine.agenerate(prompt_token_ids)
resp.output_token_ids   # 模型采样出的 token，精确
resp.output_logprobs    # 每个 token 的 logprob，精确
# 直接拼成训练样本：prompt + response，loss_mask 你说了算

token 级数据是采样的副产品，天然就有，分词也由你统一掌控。

你只能把现成 agent（Claude Code / Codex…）跑起来观察。四件事全被打破：

不控制循环
agent 自己决定发什么 prompt、调什么工具、走几轮，你无法直接命令模型生成。
说方言
agent 发的是 Anthropic / OpenAI / Google 的高层 messages，不是 token；分词在推理服务器内部发生。
多轮
一次任务 = 多次独立 LLM 调用，要拼回一条连续的训练序列。
会失败
超时、报错、截断、空输出都可能发生，还不能破坏 GRPO 组结构。

下面六节，就是 Polar 逐一解决这四点的具体算法。

01跑它 + proxy 拦截

透明代理：一个 catch-all 端点

agent 的 SDK 以为在调真实 API。Polar 把环境变量里的 base_url 指向网关，于是所有 LLM 调用都落到一个万能 POST 端点上被拦截。算法很直接：识别方言 → 翻成 OpenAI → 调 SGLang → 捕获记录 → 翻回原方言。

# 精简自 gateway/server.py:proxy_request（catch-all）
@app.api_route("/{path:path}", methods=["POST"])
async def proxy_request(request, path):
    body      = await request.json()
    api_type  = detect(full_path, headers, body)        # 识别方言: anthropic / openai_* / google
    session_id = _resolve_session_id(headers, body)        # API key 即 session_id，给请求打标签
    transformer = state.transform_manager.get(api_type)
    openai_request = transformer.transform_request(body)   # 方言 → OpenAI Chat 规范
    openai_request["model"] = state.node.model_served   # 保留原请求名，实际换成 served 模型
    # 非流式路径
    response = await state.sglang.completion(openai_request)
    state.storage.save_message(session_id, openai_request, response, ...)  # 捕获 → CompletionRecord
    return JSONResponse(transformer.transform_response(response, original_request))  # 翻回原方言
流式的悖论：很多 agent 要流式，但训练需要完整回复才能捕获全部 token。解法：对 SGLang 强制发非流式（stream=False）拿到完整结果，再切成分块逐个回吐给 agent，对端无感。

02取精确 token 与 logprob

TITO：不重新分词，向服务器要 token

这是整条流程最核心的一点。训练用的 token 必须等于采样时模型实际见过、产出的 token。若拿 agent 看到的文本重新分词，BPE 会漂移，训练分布就偏离采样分布，梯度是错的。Polar 因此直接从推理服务器的返回里取 token id 与 logprob。

# 精简自 trajectory/builder/record_utils.py:build_trace_from_completion
prompt_ids = first_choice.get("input_token_ids") or response.get("prompt_token_ids")
response_ids = _extract_response_ids(response, first_choice)
return Trace(
    prompt_ids       = prompt_ids,
    response_ids     = response_ids,
    loss_mask        = [1] * len(response_ids),        # 采样出的 token 默认全部可训练
    response_logprobs= _extract_response_logprobs(first_choice),  # logprobs.content[]
    finish_reason    = first_choice.get("finish_reason"),
)

# 取 response token：优先 token_ids，否则从 logprobs 的 token_id 重建
def _extract_response_ids(response, choice):
    token_ids = choice.get("token_ids", response.get("token_ids"))
    if isinstance(token_ids, list): return token_ids
    content = choice["logprobs"]["content"]              # 兜底
    return [item["token_id"] for item in content]
为什么要给 SGLang 打补丁：标准 OpenAI 接口不稳定返回 input_token_ids 和完整的逐 token logprob。scripts/patch/patch_sglang.sh 让服务器在回复里带上 prompt token ids 与 每个生成 token 的 logprob，TITO 才成立。

03多轮拼接 · prefix-merging

把 N 次调用拼回一条序列，且不引入分词漂移

一次 agent 运行发了多次请求。第 i+1 次的 prompt 通常恰好是「第 i 次的 prompt + 模型回复 + 工具结果等插入段」。prefix-merging 用严格的 token 前缀不变式把它们拼成一条，且对每一段精确区分"采样体"与"插入段"。

交互示例：逐步拼接一条多轮轨迹

两轮对话：第 1 轮模型采样出 a₁ a₂ ⟨eot⟩ 后调用工具；第 2 轮在工具结果后采样出 b₁ ⟨eot⟩。点"下一步"观察合并流如何长出来。

下一步 › ↺ 重置
prompt（不训练） 采样体（loss_mask=1，真实 logprob） 插入段（loss_mask=0，规范分词） ⟨eot⟩ 采样结束符（训练，兼作切分点）

算法两阶段

阶段一 · 分组

用消息级 key 做 O(1) 索引找候选链，再用严格 token 前缀检查定夺：C_{k+1}.prompt_ids 必须以 C_k.prompt_ids 开头。不满足就另起一条链，绝不污染已有链。

阶段二 · 拼接

采样体取原始 response_ids（真实 logprob，绝不 decode→encode）；插入段取下一次 prompt_ids 的规范分词，用 ⟨eot⟩ 定位切分点，loss_mask=0。

# 精简自 prefix_merging.py:_finalize_chain（拼接核心）
canonical_tail = Ci.prompt_ids[len(prev_prompt_ids):]      # = [前一轮助手体 + 插入段] 的规范分词
interstitial   = slice_interstitial(canonical_tail, prev_raw_response, eot_id)  # 找 ⟨eot⟩，其后即插入段
stream_ids += interstitial;       loss_mask += [0]*len(interstitial)   # 插入段：不训练
stream_ids += Ci.response_ids;    loss_mask += [1]*len(Ci.response_ids)  # 采样体：训练 + 真实 logprob

04只在采样 token 上算梯度

loss_mask 与 logprob 的强校验

合并流里只有 agent 真正采样出的助手 token 该进梯度。这个 0/1 不是靠"看角色 / 匹配文本"猜的，而是由 token 流结构直接确定的，分三层生成。

loss_mask 的三层生成谱系
① 诞生
每条 completion 归一化时，response 的每个 token 默认置 1（[1]*len(response_ids)）。语义：从服务器取回的 response token 就是模型采样出的，理应全可训练。prompt 不在 mask 范围内。
② 拼接重写
prefix-merging 逐段拼：采样体段沿用该 completion 的 mask（全 1）；插入段（工具结果 + 模板 glue，取自下一次 prompt 的规范分词，用 ⟨eot⟩ 切出）显式置 0。
③ 兜底校验
转 Sample 时：失败 / 中止轨迹整条清零；强校验 mask 长度 = response 长度、可训练位必有 logprob；Trace 模型层再校验值只能是 0 / 1。
# ① record_utils.py:74 — 诞生：采样出的 token 默认全部可训练
loss_mask = [1] * len(response_ids)

# ② prefix_merging.py:_finalize_chain — 拼接时逐段写
loss_mask += [0] * len(interstitial)        # 插入段：工具结果/模板 glue，不训练
loss_mask += [1] * len(Ci.response_ids)     # 采样体：模型真正生成，训练

05状态与占位

采样会失败，但 GRPO 组不能塌

一组是同一道题的 num_samples 次独立运行。其中任何一次都可能超时、报错、截断、空输出。Polar 用状态标记 + 全掩码占位，保证训练侧拿到的组结构始终完整。

状态映射 
# adapter._sample_status
TIMEOUT          → ABORTED
ERROR / error    → FAILED
finish=="length"  → TRUNCATED
否则             → COMPLETED

ABORTED / FAILED 的 trace 会把 loss_mask 整条清零。

占位保组

若一个 Session 的 Trace 全部不可用（空 token 或超 max_tokens 被丢），发一个全掩码占位 Sample，让该组样本数不错位，其余样本照常训练。

06组内相对优势归一化

GRPO 归一化，并把失败排除出基线

奖励由 evaluator 给出后，在训练器侧的后处理钩子里做组内归一化。同一 Session 的多个 Trace 共享 (group_index, index)，被当作一条轨迹（每组取一次奖励）。失败/中止轨迹不进均值与方差，避免"agent 出错"被当成"奖励=0"污染存活样本。

$$ \hat{A}_i=\frac{r_i-\operatorname{mean}\big(\{r_j\}_{j\in\text{valid}}\big)}{\operatorname{std}\big(\{r_j\}_{j\in\text{valid}}\big)+\varepsilon},\qquad \hat{A}_{i}=0\ \text{若 }i\ \text{失败} $$ 
# 精简自 reward_post_process.py:post_process_rewards（每组）
valid_mask = [not failed[k] for k in keys]      # 失败/中止轨迹标 False
vals       = tensor([reward[k] for k in keys])
vals = vals - vals[valid_mask].mean()                # 只用有效轨迹算均值
if std_norm:
    vals = vals / (vals[valid_mask].std() + 1e-6)    # GRPO 可选除方差
vals = vals * valid_mask                             # 失败轨迹优势归零

退化情形：每条轨迹只有一个 Trace 时，自动收敛为标准 GRPO 组归一化，无需特判。

07全流程

一段伪代码串起整条流程

从提交一组任务，到拿回可训练、已归一化的样本。

# Polar agent 采样全流程（伪代码）
for sample_i in range(num_samples):              # 一组 = 同一道题跑 num_samples 次
    runtime = prepare_runtime()                  # INIT：容器（可复用预热）
    harness = create_harness(agent_spec)         # 注入 base_url → 指向代理
    while agent_running:                          # RUNNING：跑它
        req  = agent.next_llm_call()             # agent 自己决定
        resp = proxy.intercept(req)              # 01 拦截：转 OpenAI → SGLang → 转回
        storage.save(CompletionRecord(req, resp))  # 含精确 token + logprob
    # POSTRUN：
    traces = PrefixMergingBuilder.build(session) # 02+03 TITO + 多轮拼接
    reward = evaluator.evaluate(traces)          # 跑测试等，给奖励
    for trace in traces:
        sample = to_slime_sample(trace)          # 04 loss_mask + logprob 校验
        if sample.empty: sample = dummy_placeholder()  # 05 占位保组
        group.append(sample)

rewards = group_normalize(group, exclude_failed=True)  # 06 GRPO 组归一化
return group                                    # → 训练器（tokens, loss_mask, rollout_log_probs, 归一化 reward）

一句话

把"对一个不可控、说方言、多轮、还会失败的黑盒 agent 采样"，拆解为：跑它（harness 运行）、拦截（proxy 捕获每次调用）、取精确 token（TITO，不重新分词）、拼接（prefix-merging 区分采样体与插入段）、只训采样 token（loss_mask + logprob 校验）、保住组（状态/占位 + 排除失败的组归一化）。每一步都只信推理服务器给出的精确 token，绝不在文本层重新分词，从而保证采样分布等于训练分布。

08实例

以 Claude Code 为环境，走一遍

把前面六环落到一个真实 agent 上。Claude Code 说 Anthropic 方言、要求流式、自带多轮工具调用，Polar 不改它一行代码，只靠环境变量与代理把它变成 RL 环境。

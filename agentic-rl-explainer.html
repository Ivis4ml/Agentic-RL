  • 自带推理：内置 SGLang / vLLM 远程引擎
  • 可中断 rollout：权重更新时打断生成并续算

动机

为什么要异步 RL

传统同步 RL 把"生成"和"训练"排成一条直线轮流执行。生成一批回答时，训练 GPU 全程空闲；训练时，推理 GPU 又全程空闲。对会输出很长推理链的大模型，这种空闲尤其严重，因为一批数据必须等到最长的那条输出结束才能进入训练。

训练里的"等"来自哪里，怎么把它降到最低

一个 RL 步 = 生成 rollout（慢）+ 训练（快）。同步执行时，生成期间训练 GPU 全程空转，这段空转就是 delay。它来自五个不同的源头，每个都有针对性的消除手段：

等待来源（delay 出在哪）消除手段 · 关键机制属于细节
训练要等整批生成完成（最根本的瓶颈）异步解耦：生成持续在另一组 GPU 上跑，训练从 replay buffer 取已就绪数据，不等当前批（max_head_offpolicyness>0）AReaL→ §05 · workflow_executor.py
一批里要等最慢/最长的那条输出可中断生成：权重到来即打断、换权重、续算，不为最慢样本空等（再 +12~17% 吞吐）AReaL→ §05 · update_weights:611
每条 rollout 的容器启动 + 多轮 + 评测耗时（真实 agent）四阶段分池 + runtime 预热复用 + 阶段感知调度：占 GPU 的 RUNNING 不被 INIT / POSTRUN 阻塞Polar→ §08 · dispatcher.py:31
权重同步本身的耗时（写 checkpoint 再重载）XCCL / NCCL 直传权重，免于写盘，延迟最低AReaL→ §04 · update_weights_from_distributed:666
生成总体就是比训练慢，整体被拖住生成集群独立扩展（disaggregation），按吞吐给两侧配比，使速率匹配AReaL→ §03 设计出发点
绕不开的代价："减少等待"与"控制陈旧"是一体两面。让生成跑在训练前面越多（η 越大），吞吐越高，但数据越陈旧。因此必须用 η 上界 + 解耦 PPO 目标 消化陈旧，否则朴素 PPO 在 η=4 会从 42.0 崩到 23.3（见 §05 的可拖动滑块）。一句话：减少 delay 的本质，就是让训练 GPU 永不空等、让生成不等最慢者、让 agent 的准备不挡 GPU，再用解耦目标把由此产生的陈旧消化掉。

训练 worker 更新完参数后，必须把新权重送回推理服务器，否则 rollout 还在用旧模型。AReaL 提供两条路径，由 WeightUpdateMeta.type 选择。

type = "xccl"

权重带版本号。生成的每个 token 记录它由哪个版本产生（versions 张量），这是异步算法纠偏的关键，见下一节。

AReaL · 算法核心

异步带来的两个难题，与解法

让生成与训练并行，会引入"数据陈旧"。AReaL 用两项配套技术处理它：陈旧度控制，以及解耦的 PPO 目标。

难题 1 · 数据陈旧（off-policy）

生成 rollout 的模型版本，往往落后于当前正在训练的版本。陈旧数据与最新模型之间存在分布差异，长输出尤其严重。

难题 2 · 单条轨迹跨多版本

可中断生成意味着：一条轨迹的不同片段，可能由不同模型版本产出。这违反了标准 PPO"所有动作来自同一策略"的前提。

解法 1 · 陈旧度控制（Staleness-Aware）

引入超参数 η = max_head_offpolicyness，限制生成版本与训练版本之间的最大差距。拖动滑块，看陈旧度如何影响最终成绩（数据来自 v0.3 博客表 3，模型 R1-Distill-Qwen-1.5B，AIME24 评分）。

4
朴素 PPO + 解耦目标

η = 0 时退化为同步 RL（约慢一倍，常用于调试基线）；η = 1 即此前"单步重叠"方法。可见：朴素 PPO 在陈旧度增大时迅速崩坏，而解耦目标把它稳住在接近同步的水平。配置：rollout.max_head_offpolicyness。

解法 2 · 解耦的 PPO 目标

标准 PPO 的重要性比值以"采样策略"为中心做裁剪，一旦采样策略陈旧，裁剪中心就错位。解耦目标把两个角色分开：

行为策略 (behavior)
真正用于采样、产出这条轨迹的策略。可能已陈旧。
近端策略 (proximal)
一个较近的目标策略，用来约束在线策略的更新方向。

直观写成（概念形式，权威表达见论文图示与 areal/trainer/ppo/actor.py 实现）：以近端策略为裁剪中心，再用行为策略做重要性纠偏。

$$ J_{\text{decoupled}}(\theta)=\mathbb{E}\!\left[\underbrace{\frac{\pi_{\text{prox}}}{\pi_{\text{behave}}}}_{\text{陈旧度纠偏}}\cdot\,\min\!\Big(\tfrac{\pi_\theta}{\pi_{\text{prox}}}\hat{A},\ \text{clip}\big(\tfrac{\pi_\theta}{\pi_{\text{prox}}},1-\epsilon,1+\epsilon\big)\hat{A}\Big)\right] $$

论文中该目标以图片给出（assets/figures/decoupled_ppo_obj.png），此处为便于理解的概念化写法，不作为逐符号的权威定义。AReaL 还能用"版本感知的对数线性插值"近似近端策略的对数概率，省去一次前向计算（docs/en/algorithms/prox_approx.md）。

可中断生成 · 一条轨迹如何跨版本

当 update_weights 到来，rollout worker 立即打断所有生成，丢弃旧权重算出的 KV cache，用新权重重算后继续解码未完成的序列。因此一条长轨迹天然由多个版本的片段拼成，这正是难题 2 的来源，也由解耦目标处理。

AReaL 的核心，就是这两点的配合

系统侧：可中断生成，让推理 GPU 永不空等（实现于 update_weights 触发的打断与续算）。算法侧：解耦 PPO 目标，把"可中断 + 异步"必然带来的版本错位纠正回来。二者缺一不可：只有系统不改算法，陈旧度一大就崩（见上面 η=4 的对照）；只改算法不要系统，就退回同步的低效。读懂下面三处代码，就抓住了 AReaL 的精髓：

fsdp_engine.py:611 · update_weights（触发打断与同步） actor.py:408 · grpo_loss_fn（含解耦目标） actor.py:604 · compute_prox_logp_approximations（近端策略近似）

      Polar · 架构
      
Polar：把现成 agent 当成 RL 环境

      
和 AReaL 的一体化不同，Polar 是一个解耦的 rollout 服务。中央 Rollout Server 接收任务并分发给若干 Gateway Node，每个 Gateway 在隔离 runtime 里跑真实 agent 外壳，用一个透明代理拦截 agent 发出的所有 LLM 调用，再把记录构造成训练用轨迹。点击组件查看细节。


      

        
设计出发点 · Polar 为什么这样构架

        

          
Polar 的总目标只有一句：低侵入、中立，做覆盖多种训练与推理框架的"最小公共基础"（lowest common ancestor）。它刻意不绑定任何具体训练器或 agent 实现，只在二者之间提供一层薄而稳定的服务边界。三根支柱都是从一个具体痛点推出来的：

由此得到

每个 Gateway 把一个 session 推过四个各自独立工作池的阶段。关键在于：当某个 session 的 agent 正在占用推理 GPU（RUNNING），另一个 session 可以同时在做容器初始化（INIT）或轨迹构造（POSTRUN），从而让宝贵的推理资源不空等。

        

          
stage 1

→

→

→

INIT
启动 Docker / Apptainer 容器，执行 prepare 动作，上传任务文件。完成后进入 READY。

          
READY
runtime 已就绪，等待一个"运行槽"。这是一个预热池：容器可提前装依赖、下载模型。槽位数 = max_run_workers。

          
RUNNING
agent 外壳执行，代理拦截 LLM 调用并写入磁盘。这是占用推理 GPU 的阶段，所以工作池最受约束。

          
POSTRUN
把捕获的 completion 构造成 Trace，运行评测器算奖励，回调 Rollout Server，清理容器。这是纯 CPU 工作，不阻塞 GPU。

        

        

        

          

            
Runtime Pooling 运行时复用

            
节点维护一个就绪缓冲（ready_buffer_target）。容器启动很贵，复用预热好的 runtime 可省下分钟级开销。若评测需要全新环境，还会在 agent 运行时后台预热评测 runtime。

          

          

            
阶段感知调度

            
Rollout Server 分发 session 时按多级压力打分：优先 RUNNING 压力最低的节点（避免 GPU 瓶颈），其次 POSTRUN、INIT。POSTRUN 积压过多则暂不派新活。

          

        

        
实现：四阶段调度 src/polar/gateway/dispatcher.py（SessionStage 枚举）；调度打分 src/polar/rollout/balancer.py（NodeScheduler._node_score）。

      


      
      

        
TITO = Token-In-Token-Out。训练需要的是 token 级数据，而 agent 与代理之间往来的是文本消息。要把拦截到的流量还原成可训练轨迹，代理必须从推理服务器拿到三样东西：

        

          
prompt token ids
实际喂给模型的输入 token

          
response token ids
模型实际采样出的 token

          
per-token logprobs
每个生成 token 的对数概率

        

        
标准 OpenAI 接口不稳定返回 prompt token ids，因此安装时需给 SGLang 打一个补丁（scripts/patch/patch_sglang.sh），让它在回复里带上 prompt token ids 与完整的逐 token logprobs。

        
构造分三步：

        

          
① 捕获
代理把每次调用存成 CompletionRecord，按时间序聚成 CompletionSession。

          
② 归一化
每条 completion 转成一个 Trace：抽出 prompt_ids、response_ids、logprobs，loss_mask 默认全 1（采样出的 token 都可训练）。

          
③ 前缀合并
多轮 agent 会连续发多次请求。Prefix-Merging Builder 把"后一次请求恰为前一次 prompt + 采样回复"的链条拼成一条轨迹：助手 token 用真实采样值与真实 logprob，工具结果等插入段用服务器的规范化 token 且 loss_mask = 0。

        

        
实现：src/polar/trajectory/builder/record_utils.py、prefix_merging.py；数据模型 src/polar/trajectory/models.py。

      


      
      

        
普通 RL 采样：直接调模型，prompt 进、token + logprob 出，你完全控制生成循环。对真实 agent 采样，这三点全被打破：你不控制生成循环（agent 自己决定 prompt 与工具调用）；agent 说的是高层 API 方言而非 token；一次任务是多次 LLM 调用且可能中途失败。Polar 用一套环环相扣的设计把它解决。


        

          
采样的层级结构（容易混淆，先理清）

          

            
一个 Task · 1 条 instruction

            
└─ × num_samples 个 Session · 同一个 GRPO 组（group_index）

            
└─ 每个 Session = 完整跑一遍 agent（独立 runtime）

            
└─ 多次 LLM 调用（CompletionRecord）→ builder 拼接

            
└─ 1 个或多个 Trace · 多轮 / 上下文压缩会拆成多条

            
└─ 每个 Trace → 1 个 Sample

          

          
同一个 Session 的所有 Trace 共享同一个 (group_index, index)：优势归一化时被当作一条轨迹，但每个 Trace 各自把采样出的 token 贡献进梯度。

        


        

          
TITO 一致性
最核心的采样问题。训练用的 token 必须等于采样时模型实际见过/产出的 token。Polar 绝不重新分词，直接从推理服务器取精确的 prompt_ids、response_ids 与逐 token logprob，避免分词漂移导致"训练分布 ≠ 采样分布"。

          
loss_mask
只有 agent 真正采样出的助手 token 置 1 且必须带对应 logprob；工具结果、用户轮、为拼接复制的规范化 token 置 0。校验：可训练 token 缺 logprob、或长度不匹配，直接抛 RolloutLogprobError。

          
失败 / 截断状态
超时 → ABORTED，报错 → FAILED，finish_reason==length → TRUNCATED，否则 COMPLETED。失败/中止的 trace 把 loss_mask 清零。

          
保持组完整
若一个 Session 的 Trace 全部不可用，发一个全掩码占位 Sample（remove_sample=True），让该组样本数不错位，其余样本照常训练。

          
排除失败基线
组内归一化时，失败/中止轨迹被排除出均值与方差，不污染存活样本的优势估计。

          
采样 × 权重更新
更新窗口内 bridge pause_admission() 并调 /admin/sglang/pause 暂停生成，更新后 resume。这是请求边界上的暂停，对应但不同于 AReaL 引擎内的可中断生成。

流式的悖论：agent 要流式，训练要完整回复

组内相对优势归一化在哪里发生

      
GRPO 的组内归一化由 Slime 侧的奖励后处理钩子完成（src/slime_bridge/reward_post_process.py），在收齐所有样本后、训练前执行：

      

        $$ \hat{A}_i=\frac{r_i-\operatorname{mean}\big(\{r_j\}_{j\in \text{valid}}\big)}{\operatorname{std}\big(\{r_j\}_{j\in \text{valid}}\big)+\varepsilon},\qquad \text{失败/中止的轨迹被排除出基线} $$
      

      
失败或中止（FAILED / ABORTED）的轨迹不计入组内均值与方差，避免 agent 出错污染存活样本的优势估计；归一化后的奖励再广播回该轨迹的所有 Trace。若每条轨迹只有一条 Trace，则退化为标准 GRPO 组归一化。


      

        对照 AReaL：同样是 GRPO 组归一化，AReaL 在自己的训练循环里用 NormConfig 完成（见第 06 节）；Polar 则把它放在训练器侧的后处理钩子里。这正体现了两者的设计差异：一体化内建 vs 解耦交给训练器。
